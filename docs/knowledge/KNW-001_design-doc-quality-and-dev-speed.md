## 背景

たすきば Knowledge Relay の MVP-1a 開発において、設計書（DESIGN.md）を詳細に作り込んでから実装に着手した。設計書には ER 図、テーブル定義、API エンドポイント一覧、権限マトリクス、状態遷移図、バリデーションルール、セキュリティ設計が網羅的に記載されていた。

この設計書をもとに Claude Code で実装を行ったところ、当初 16 日間を見込んでいた MVP-1a の全 7 タスクが 1 日で完了した。

## 内容

### 計画と実績の比較

| 項目 | 計画 | 実績 | 差分 |
|---|---|---|---|
| 開発期間 | 04/15〜05/01（16日間） | 04/15（1日） | 15日前倒し |
| タスク数 | 7タスク | 7タスク + テスト104件 | 計画以上 |
| テスト | リリース条件として定義 | 104件作成・全通過 | 計画外の追加成果 |

### 前倒しの要因分析

| 要因 | 詳細 | 寄与度 |
|---|---|---|
| **AI 支援開発（Claude Code）** | コード生成速度が人手の数十倍。ビルドエラーも数分で修正 | 大 |
| **設計書の完成度** | 実装時の判断コストがほぼゼロ。設計書をそのままコードに変換できた | 大 |
| **設計書と実装の対応関係の明確さ** | 設計書のどのセクションがどのファイルに対応するかが明確 | 中 |

### 設計書の完成度が効いた具体的な箇所

| 設計書のセクション | 実装への効果 |
|---|---|
| ER 図 + テーブル定義（セクション 4〜5） | Prisma Schema をほぼ機械的に生成。カラム名・型・制約を迷わず決定 |
| API エンドポイント一覧（セクション 7） | Route Handler のファイル構造・HTTP メソッド・レスポンス形式が確定 |
| 権限マトリクス（セクション 8） | checkPermission 関数のロール別許可アクション定義をそのまま実装 |
| 状態遷移図（セクション 6） | State Machine のルール定義をそのまま配列化 |
| バリデーションルール（仕様書） | Zod スキーマの min/max/enum 値を仕様書からそのまま転記 |
| セキュリティ設計（セクション 9） | IDOR 防止パターン、監査ログ記録箇所、機密フィールド除外リストを実装時に参照 |

### 設計書に記載されていなかったことで実装時に判断が必要だった箇所

| 箇所 | 内容 | 対処 |
|---|---|---|
| Prisma 7 の破壊的変更 | datasource URL の設定方式が変更されていた | 公式ドキュメントを参照して対応 |
| shadcn/ui v4 の API 変更 | asChild プロパティの廃止、Select の onValueChange が null 許容に | ビルドエラーから判明し修正 |
| Next.js 16 の middleware 非推奨警告 | middleware → proxy への移行が推奨 | 現時点では middleware を維持（動作に支障なし） |

## 結論

設計書の完成度は、AI 支援開発において特に開発速度に直結する。

人間の開発者が設計書を参照する場合、設計書に曖昧さがあっても経験や直感で補完できる。しかし、AI（Claude Code）は設計書に書かれた情報をそのまま実装に変換するため、設計書が曖昧だと判断を求める回数が増え、速度が低下する。逆に、設計書が詳細であるほど AI の変換効率が最大化される。

この「設計書の完成度 → AI の変換効率 → 開発速度」の関係は、今後の開発でも適用可能な法則である。

## 推奨事項

### 今後のプロジェクトへの適用

1. **設計書に以下を必ず含める**
   - ER 図 + 全テーブルのカラム定義（型・制約・デフォルト値を含む）
   - API エンドポイント一覧（HTTP メソッド・パス・ロール・レスポンス形式）
   - 権限マトリクス（ロール x 操作 x 状態の3次元）
   - バリデーションルール（必須/任意・文字数制限・enum値）
   - 状態遷移図（遷移条件を含む）

2. **設計書のレビュー基準**
   - 「この設計書だけで、実装時に判断が不要なレベルか？」を基準とする
   - 曖昧な箇所は実装前に解決する（実装中の判断はコストが高い）

3. **設計書とコードの対応関係を明記する**
   - 設計書のセクション番号をコードのコメントに残す（例: `// 設計書: DESIGN.md セクション 6`）
   - 設計書の変更時にコードへの影響範囲を特定しやすくする

### AI 支援開発の効率を最大化するために

- 設計書は「人間が読む」だけでなく「AI が処理する」前提で書く
- 曖昧な表現（「適切に」「必要に応じて」）を避け、具体的な値・条件を明記する
- マスタデータの enum 値は設計書に全列挙する
