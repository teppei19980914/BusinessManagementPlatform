#### WBS テンプレート仕様 (旧フロー: 別プロジェクトへの雛形流用)
- 形式: CSV（レベル列と行順序で階層構造を表現、Excel で編集可能）
- CSV列: レベル, 種別(WP/ACT), 名称, WBS番号, 予定開始日, 予定終了日, 見積工数, 優先度, マイルストーン, 備考
- エクスポート対象: チェックボックスで選択したタスク（未選択時は全タスク）
- エクスポート時: 担当者はリセット（テンプレートとして再利用可能）、BOM付きUTF-8でExcel対応
- インポート時: 進捗・ステータスは初期状態（`not_started` / `0%`）で作成
- 上限: 500件/テンプレート
- インポートバリデーション: 親参照整合性（存在しない親の検出）、tempId 重複、循環参照、親種別チェック
- エラー時: トランザクションにより全タスクがロールバック（不正データは残らない）

#### WBS 上書きインポート仕様 (feat/wbs-overwrite-import / 新規追加)

**目的**: 同一プロジェクト内の WBS を 「export → Excel 編集 → re-import」の往復編集サイクルで管理可能にする。旧テンプレート流用フローとは別系統で並走する。

**CSV フォーマット (新形式、17 列)**:

| # | 列名 | 必須/任意 | 区分 | import 動作 |
|---|---|---|---|---|
| 1 | ID | 任意 | 突合 | UUID。値あり=既存と突合し UPDATE、空欄=新規作成 |
| 2 | レベル | 必須 | 階層 | 1〜N、行順序と組合せで階層を再構築 |
| 3 | 種別 | 必須 | 階層 | WP / ACT。WP↔ACT 切替は dry-run でエラー扱い |
| 4 | 名称 | 必須 | 計画 | UPDATE 反映 |
| 5 | WBS番号 | 任意 | 計画 | UPDATE 反映 |
| 6 | 担当者氏名 | 任意 | 計画 | ProjectMember 内で氏名一意 lookup、複数該当 / 不在=エラー |
| 7 | 予定開始日 | 任意 | 計画 | UPDATE 反映 (ACT のみ。WP は子から自動算出のため CSV 側を無視) |
| 8 | 予定終了日 | 任意 | 計画 | UPDATE 反映 (ACT のみ) |
| 9 | 見積工数 | 任意 | 計画 | UPDATE 反映 (ACT のみ) |
| 10 | 優先度 | 任意 | 計画 | `low` / `medium` / `high` |
| 11 | マイルストーン | 任意 | 計画 | `○` / 空 |
| 12 | 備考 | 任意 | 計画 | UPDATE 反映 |
| 13 | ステータス ※read-only | - | 進捗 | import 時は無視 (DB と異なる場合は dry-run で warn) |
| 14 | 進捗率 ※read-only | - | 進捗 | 同上 |
| 15 | 実績工数 ※read-only | - | 進捗 | 同上 |
| 16 | 実績開始日 ※read-only | - | 進捗 | 同上 |
| 17 | 実績終了日 ※read-only | - | 進捗 | 同上 |

ヘッダー行に `※read-only (import 時無視)` の注記を 13〜17 列に明示する。BOM 付き UTF-8 で出力 (Excel 対応)。

**エクスポート仕様 (新形式)**:
- エンドポイント: 既存 `POST /api/projects/[id]/tasks/export` を拡張 (後方互換維持のため `mode='sync'` パラメータで新形式を返す)
- 担当者・進捗・実績はリセットせず DB 値そのまま出力
- 画面上「IDを表示」トグル ON 時の表示と整合する

**インポートフロー (2 ステップ)**:

1. **dry-run プレビュー**: `POST /api/projects/[id]/tasks/sync-import?dryRun=1`
   - レスポンスにサマリ ({ added, updated, removed, blockedErrors, warnings }) と行ごとの差分を返す
   - 削除候補のうち進捗を持つタスク (progressRate > 0 or actualStartDate != null) は赤エラー扱い
   - DB は変更しない

2. **本実行**: `POST /api/projects/[id]/tasks/sync-import` (dryRun なし)
   - body に `removeMode: 'keep' | 'warn' | 'delete'` を含める
   - 成功 → import 件数サマリを返す
   - 失敗 → インポート前の状態に完全復元 (audit_log の beforeValue から)

**画面操作**:
- タスク一覧画面に「IDを表示」トグルボタンを追加 (既定 OFF、ON で先頭に ID 列表示)
- 「WBSをエクスポート」「WBSを上書きインポート」ボタンを別途追加 (PM/TL + admin のみ表示)
- 上書きインポートは dry-run プレビューダイアログを必ず経由する 2 ステップ UX
  - サマリ + 行ごと差分テーブル + 削除候補セクション (赤強調)
  - 「削除モード」ラジオボタン (保持 / 警告のみ / 削除実行)
  - 「確定実行」ボタンで本 API 呼び出し

**エラー分類**:

| 重大度 | 条件 | 動作 |
|---|---|---|
| ブロッカー | ヘッダー不正 / 必須列空 / 種別不正 / 親不在 / WP↔ACT 切替 / ID 不一致だが名称一致 / 担当者氏名がメンバー外 or 複数該当 / CSV 内 ID 重複 / CSV 内同階層名称重複 / 循環参照 / 進捗あり削除候補 (削除モード時) | dry-run でエラー、実行不可 |
| 警告 | 進捗系列の値が DB と異なる / 削除候補が存在する (warn モード) / read-only 列を編集している | 実行可、確認後 OK |
| 情報 | 追加 / 更新 / 名称変更 / 親変更 | 表示のみ |

**バリデーション + ロールバック**:
- PgBouncer 制約で `prisma.$transaction` 不可のため、開始時に対象タスクの完全コピーを `audit_log.beforeValue` に保存
- エラー発生時:
  1. それまで作成済の Task は物理削除
  2. UPDATE 済の Task は beforeValue から完全列復元
  3. DELETE 済の Task (deletedAt セット済) は `deletedAt: null` に戻す
- 完了後: `audit_log` を 1 件追加 (action='SYNC_IMPORT', afterValue=summary)

**上限**: 500 件/インポート (テンプレート版と同等)

- 列名のヘッダーゆらぎ吸収
- Undo (実行後の取り消し)
- 進捗系列を書き戻し可能にする上級モード

## 提案型サービス v1 画面仕様 (PR #65)

## 16. 核心機能: 提案型サービス (PR #65)

### 16.1 機能概要

本プロダクトの核心価値である「**過去の資産を未来のプロジェクトに自動で活かす**」機能。
人手で過去資産を探す限り発生する抜け漏れを極小化し、未然に防げた課題の顕在化を減らす。
技術詳細は DESIGN.md §23 を参照。

### 16.2 対象資産

| 資産 | 絞り込み | 理由 |
|---|---|---|
| ナレッジ (Knowledge) | `visibility='public'` + **自プロジェクト未紐付け** (PR #160) | 全社共有済の知見のみ候補化。自プロジェクトに紐付け済のナレッジは「参考」として並べる意味がない (Issue/Retrospective と parity) |
| 過去の課題 (Issue) | `type='issue'` + `state='resolved'` + 他プロジェクト | リスクは未発生のため除外、解消済課題のみ過去資産として扱う |

### 16.3 画面・操作

| 画面 / タイミング | 動作 |
|---|---|
| 新規プロジェクト作成直後 | `/projects/[id]?suggestions=1` に遷移 → 提案モーダルが自動起動 |
| プロジェクト詳細「参考」タブ | 常時参照可能、タブ切替で即座に提案リスト表示 |

### 16.4 採用操作

| 対象 | ボタン | 挙動 |
|---|---|---|
| Knowledge | 「このプロジェクトに紐付け」 | `knowledge_projects` に中間レコード追加。採用直後はその場で「紐付け済」バッジに切替、再フェッチ時は提案候補から除外される (PR #160) |
| 過去 Issue | 「雛形として採用」 | 新規 Issue を type='issue' / state='未対応' / visibility='下書き' で複製。起票者は採用者に差し替え、`result`・`lessonLearned` は継承しない |

### 16.5 権限

| ロール | 閲覧 (GET) | 採用 (POST) |
|---|---|---|
| システム管理者 | 全プロジェクト | 全プロジェクト |
| プロジェクト PM/TL / member | 自プロジェクト | 自プロジェクト |
| 非メンバー | 不可 | 不可 |

### 16.6 精度指標 (内部表示)

提案 1 件ごとに以下の 3 つのスコアを保持し、tooltip で根拠を表示する。
- `tagScore`: タグ Jaccard 係数 (0〜1)
- `textScore`: pg_trgm テキスト類似度 (0〜1)
- `score`: 上記の重み付き平均 (既定 50%/50%)

### 16.7 プロジェクトタグの入力仕様

新規プロジェクト作成時に以下の 3 種類のタグをカンマ区切りで入力できる。
提案精度向上のため入力を推奨する。

| タグ種別 | 例 | マッチング対象 |
|---|---|---|
| 業務ドメイン | 金融 / 基幹業務 / 会計 | Knowledge.businessDomainTags (Phase 2 で対応完了) |
| 技術スタック | React / Next.js / TypeScript / PostgreSQL | Knowledge.techTags |
| 工程 | 要件定義 / 設計 / 開発 / 試験 | Knowledge.processTags |

### 16.8 Phase 2 で追加された機能 (同 PR #65)

**過去振り返りの提示**: プロジェクト詳細「参考」タブ + 新規作成後モーダルに、類似する過去振り返りの問題点・次回事項を参考情報として表示 (採用操作なし)。

**ナレッジ作成時のタグ入力**: ナレッジ作成フォームに 3 種類のタグ入力欄 (業務ドメイン / 技術スタック / 工程) を追加。将来のプロジェクトからの類似検索精度を上げる。

**リスク起票時の inline 提案**: リスク/課題 起票ダイアログで、件名・内容を入力中に 500ms debounce で類似する過去課題 (他プロジェクト) を検索し、警告バナーとして表示。「過去に同種の事象があるので確認を」と気付きを与える UX。

## 17. 一覧列幅のドラッグリサイズ (PR #68)

### 17.1 目的

各一覧/全〇〇画面で長文列 (タスク名 / 件名 / 顧客名 等) が切り詰められ、ユーザが中身を確認できない問題を **UI 上のドラッグ操作**で解消する。

### 17.2 操作

- 列ヘッダの右端 (6px 幅) にカーソルを近づけると `col-resize` カーソルに変化
- マウスダウン + ドラッグで任意の幅に変更
- 各一覧の右上にある「列幅をリセット」ボタンで既定値に戻す

### 17.3 対象画面

| 画面 | 対応 |
|---|---|
| `/projects` | 全列リサイズ可 |
| `/risks` / `/issues` | 全列リサイズ可 |
| `/retrospectives` | 全列リサイズ可 |
| `/knowledge` | 全列リサイズ可 |
| プロジェクト詳細: リスク/課題/WBS 一覧 | 全列リサイズ可 |
| `/my-tasks` | 全列リサイズ可 |
| プロジェクト詳細: ガント | **タスク名列のみ**リサイズ可 (日付列は固定) |

### 17.4 永続化ポリシー

- 幅は **sessionStorage** (`resizable-cols:<tableKey>` / `gantt:<projectId>:name-col-width`) に保存
- **タブ/ブラウザを閉じると既定値に戻る** (既存の折りたたみ状態・フィルタと同じ方針)

## タスクの状況 / 進捗率 整合性ルール (PR #69 Task 1)

## 19. タスクの状況 / 進捗率 整合性ルール (PR #69 Task 1)

### 19.1 既存ルール (PR #39 から)

- **状況 = 完了 → 進捗率 = 100 を強制**
  - UI のステータス変更時に進捗率を 100 に自動補正
  - サーバ側 (updateTask / bulkUpdateTasks / updateTaskProgress) でも同様に正規化

### 19.2 追加ルール (PR #69)

- **進捗率 = 100 → 状況 = 完了 を強制** (逆方向も双方向に整合)
  - UI で進捗率を 100% に入力した時点で状況が自動で「完了」に切り替わる
  - サーバ側でも同ルールを検証/強制 (不正な "進捗 100% + 状況 = 進行中" 等の組合せを防止)

### 19.3 競合時の優先順位

同一リクエスト内で status と progressRate の両方が矛盾して指定された場合:
- `progressRate=100` と `status≠completed` の組 → **「進捗 100% ならば完了」を優先** (業務ルールとして矛盾する組合せを許容しない)
- `progressRate<100` と `status=completed` の組 → 既存通り「完了 → 進捗 100%」を優先

## 21. 個人メモ機能 (PR #70)

### 21.1 目的

各アカウントが業務上調査して得た知見・作業メモを保存する個人ストック。
既存の Knowledge はプロジェクト紐付け必須で共有前提だったため、
「**プロジェクトに紐付かない個人の知見置き場**」を新設する。

### 21.2 公開範囲 (visibility)

| 値 | 挙動 |
|---|---|
| `private` (既定) | 作成者のみ閲覧可。admin も含め他者は完全に閲覧不可 |
| `public` | 全メモ画面に公開、全ログインユーザが閲覧可 (編集/削除は作成者のみ) |

### 21.3 画面 (PR #71 で 2 画面構成に分離)

| 画面 | URL | 導線 | 表示内容 | 操作 |
|---|---|---|---|---|
| **メモ** (個人管理) | `/memos` | 右上ユーザ名プルダウン → メモ | 自分のメモのみ (private/public 両方) | 作成・編集・削除すべて可能 |
| **全メモ** (横断表示) | `/all-memos` | 上部ナビ (プロジェクト一覧と同列) | 全ユーザの public メモ (自分の公開メモを含む) | 行クリックで**参照専用**ダイアログ (read-only) |

- 両画面とも列幅はリサイズ可 (PR #68 機構、各画面で tableKey を分離)
- 全メモ画面で開く詳細ダイアログは `fieldset disabled` で全入力を読み取り専用化

### 21.4 操作

- **作成**: `/memos` 画面の右上「メモ作成」ボタン (ダイアログ)。全メモ画面には作成ボタンなし
- **編集/削除**: `/memos` 画面でのみ可能 (行クリック or 操作列ボタン)
- **URL 添付**: PR #64 の仕組みを拡張 (entityType='memo')。作成時に staging、編集時に即 CRUD。全メモ画面では閲覧のみ

### 21.5 タグなし

業務知見として有用かは人間の目で判断するため、Knowledge のような tech/process/businessDomain タグは持たせない。
(将来 Knowledge 昇格機能で格上げ時に必要なら別 PR で追加)

### 21.6 認可

- **閲覧**: 作成者 OR visibility='public' のユーザ。**admin 特権なし**
- **作成**: 認証済みユーザなら誰でも
- **編集/削除**: 作成者のみ。**admin 特権なし**
- 他人の private メモにアクセスしようとしても「存在しない」扱い (404) で情報漏洩を防止

### 21.7 API

| メソッド | パス | 認可 | 備考 |
|---|---|---|---|
| GET | /api/memos?scope=mine | 認証済みユーザ | 既定。自分のメモのみ |
| GET | /api/memos?scope=public | 認証済みユーザ | 全ユーザの public メモ (PR #71) |
| POST | /api/memos | 認証済みユーザ | |
| GET | /api/memos/:id | 閲覧権限あるユーザのみ (本人 or public) | |
| PATCH | /api/memos/:id | 作成者のみ | |
| DELETE | /api/memos/:id | 作成者のみ | |

### 21.8 添付 (PR #64 連携)

- `entityType='memo'` を ATTACHMENT_ENTITY_TYPES に追加
- memo は project スコープ外のため、attachment の認可は project membership ではなく memo.userId + visibility で判定
- `authorizeMemoAttachment(memoId, viewerUserId, mode)` を新設 (read/write で権限が異なる)
- batch 取得 API は閲覧可能 memo の ID にフィルタしてから attachments を返す (URL 漏洩防止)

## 日付項目の入力効率化 (PR #71 追加)

## 22. 日付項目の入力効率化 (PR #71 追加)

### 22.1 変更内容

全ての日付入力項目 (予定開始日 / 終了日・実施日・期限 等) の**右隣**に「**今日**」「**削除**」ボタンを常時表示。

| ボタン | 動作 |
|---|---|
| 今日 | 1 クリックで今日日付 (ローカルタイム YYYY-MM-DD) をセット |
| 削除 | 1 クリックで値をクリア (空欄化)。未入力時は disabled |

必須項目 (required) では「削除」ボタンを非表示にする (空にできない項目で混乱を招かないため)。

### 22.2 導入画面

- プロジェクト新規作成 / 編集 (開始・終了予定日)
- WBS 新規作成 / 個別編集 / bulk edit / bulk actual (予定・実績の各日付)
- 振り返り作成 / 編集 (実施日)
- リスク/課題 編集 (期限)

### 22.3 背景

ブラウザ標準 `<input type="date">` では「今日」「削除」相当操作がカレンダーポップオーバー内部にしかなく、2 ステップ操作 (開く → 探す) が必要だった。ボタン常時露出により 1 ステップ化。

### 22.4 自作カレンダーへの置換 (PR #72)

右側に「今日」「削除」ボタンを外出ししたことで、カレンダーポップアップ内部のベンダー製
「今日」「クリア」ボタンは役割が重複し、UI の一貫性を損ねていた。しかし native `<input type="date">`
ではそれらを CSS / JS で非表示にできない。

よって PR #72 で native input を廃止し、自作の Popover + 日付グリッドに置換した:

- **カレンダー内には月ナビと日付グリッドのみ** (今日 / クリアは出現しない)
- 外側の「今日」「削除」ボタンは PR #71 と同じ仕様で維持
- value / onChange は引き続き `YYYY-MM-DD` 文字列 (呼び出し側 11 箇所に変更なし)

## 言語・タイムゾーン設定 (PR #118/#119/#120 追加)

## 24. 言語・タイムゾーン設定 (PR #118/#119/#120 追加)

### 24.1 設定画面での選択肢

**設定 → 言語・タイムゾーン**に 2 つのセレクトボックスを配置:

- **言語**: `SUPPORTED_LOCALES` (`src/config/i18n.ts`) のキー + 「システム既定を使用」
  - 現在 **選択可能** (`SELECTABLE_LOCALES=true`): `ja-JP` のみ
  - 表示するが **選択不可** (`SELECTABLE_LOCALES=false`): `en-US` (※準備中、PR #120 で制御)
    - UI: `<option disabled>` + 「※準備中」表記
    - API: `isSelectableLocale` で 400 拒否 (UI 迂回 / curl 直叩き防止)
  - 後続 PR で `src/i18n/messages/en-US.json` + UI 文字列抽出 + 英訳が完了次第、`SELECTABLE_LOCALES['en-US']` を true に切り替えて選択可にする
- **タイムゾーン**: `Intl.supportedValuesOf('timeZone')` で動的生成した IANA 名一覧 + 「システム既定を使用」
  - ブラウザ非対応時は代表値 (UTC, Asia/Tokyo, America/New_York 等) の fallback

### 24.2 既定値と永続化

- 既定は両方とも「システム既定を使用」(= null 保存) → `src/config/i18n.ts` の `DEFAULT_TIMEZONE` / `DEFAULT_LOCALE` (env で上書き可) が適用
- ユーザが変更した値は `users.timezone` / `users.locale` に保存 (PR #118 マイグレーション済)
- null へ戻すことも可能 (UI 上「システム既定を使用」を選択)

### 24.3 反映タイミング

1. 設定画面で選択 → 「保存」押下
2. `PATCH /api/settings/i18n` で DB 更新
3. `useSession().update({ timezone, locale })` で JWT 再発行
4. `router.refresh()` でサーバコンポーネント再描画 + `useFormatters()` が新値を反映
5. 全画面の日時表記が即時切り替わる (`2026/04/24 09:00` → `04/24/2026 00:00` 等)

### 24.4 データ格納方針

- **DB**: 常に UTC (`timestamptz`) で格納 / API 入出力は ISO 8601 UTC (`...Z`)
- **描画**: `formatDate` / `formatDateTime` / `formatDateTimeFull` ヘルパ経由で描画時に解決
- **入力**: 現時点はユーザ TZ で解釈する実装 (date-picker 入力時) は **未整備** (PR #121 予定)

### 24.5 セキュリティ

- API サーバは以下のみ受理 (400 VALIDATION_ERROR で拒否):
  - `timezone`: `Intl.DateTimeFormat` が受理する IANA 名、または `null`
  - `locale`: **`SELECTABLE_LOCALES=true` のキー** (PR #120 で厳格化)、または `null`
    - `SUPPORTED_LOCALES` に含まれていても `SELECTABLE_LOCALES=false` (例 `en-US`) は拒否
    - 既に DB に `en-US` が入っている可能性は無いが、将来的にロケールを削除する場合も
      `isSupportedLocale` はフォールバックが effect せず `resolveLocale` は FALLBACK ロケールに戻る設計
- 任意文字列による DB 汚染 / 未整備ロケールへの誘導を防ぐ (UI disabled との多層防御)
- 本人のみ更新可 (`/api/settings/i18n` は `getAuthenticatedUser` のみ要件、他ユーザ設定は触れない)
