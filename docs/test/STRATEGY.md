# テスト戦略 — 自動化と手動テストの役割分担

- 作成日: 2026-04-22
- 対象: PR #90 以降で整備した自動テスト体制 + 人間が引き続き担うべき領域

## 1. 全体像

### 自動化 3 層

| 層 | ツール | 対象 | 実行タイミング |
|---|---|---|---|
| **ユニットテスト** | Vitest | 関数・コンポーネント単位の機能 | commit push 毎 (CI) |
| **E2E テスト** | Playwright | ブラウザ越しのユーザフロー | PR push 毎 (CI) + 日次 cron |
| **視覚回帰テスト** | Playwright toHaveScreenshot | UI の見栄え (pixel 比較) | PR push 毎 (CI) |

### 役割分担

```
┌──────────────────────────────────────────────────────────┐
│ 人間の判断が必要な領域                                    │
│ (UAT / a11y / 互換性 / 性能 / 運用 / セキュリティ深層)  │
│                                                          │
│  ┌───────────────────────────────────────────────────┐  │
│  │ 自動化された領域 (PR #90 以降で整備・累積)            │  │
│  │  ユニット + E2E + 視覚回帰                          │  │
│  │  → 機能仕様の回帰検知・デグレ検出                    │  │
│  └───────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**自動テストが green = 「基準通り動作している」**
**人間テストが通る = 「基準自体が正しく、ユーザにとって価値がある」**

## 2. 自動化で **カバーできない領域**

以下は自動テストの範囲外。人間の目・手・判断が必要。

### 2.1 UAT (User Acceptance Testing) / 業務適合性

- 「この機能は実業務で使いやすいか?」
- 「ワークフローは業務プロセスに合うか?」
- 「画面遷移の "感覚" が違和感ないか?」

E2E は「仕様通りに動く」ことを保証するが、**仕様自体が正しいか** は業務担当者が使って判断する領域。試験運用フェーズで必ず通す。

### 2.2 クロスブラウザ / クロスデバイス

**自動テストの範囲**:
- Chromium のみ (Playwright の `projects: [{ name: 'chromium' }]`)
- 1440×900 固定解像度

**人間が確認すべき**:
- Firefox / Safari / Edge (Chromium 以外のレンダリング差異)
- モバイル (iPhone / Android)
- タブレット (iPad)
- 画面縮小 (1280 / 1024 / 768 等) でのレイアウト崩れ
- 高解像度 (4K) での文字サイズ感

### 2.3 アクセシビリティ (a11y) の実地確認

**自動テストがカバー**:
- `htmlFor` / `aria-label` / `aria-checked` 等、ARIA 属性の存在 (getByRole 経由で間接チェック)

**人間が確認すべき**:
- キーボード操作のみで全機能を使えるか (Tab / Enter / Esc / 矢印キー)
- スクリーンリーダー (NVDA / VoiceOver / JAWS) での読み上げの自然さ
- カラーコントラスト比 (WCAG AA: 4.5:1 以上)
- フォーカスリングの視認性
- アニメーション停止設定 (prefers-reduced-motion) の尊重

### 2.4 性能 / ロード

**自動テストがカバー**:
- 単体テストの実行時間監視 (CI タイムアウト)
- E2E 各 step のタイムアウト (10s)

**人間が確認すべき**:
- 実データ規模 (1000 プロジェクト / 10000 タスク等) での応答速度
- 同時接続 (複数ユーザが同時操作) 時の挙動
- Lighthouse スコア (Performance / Accessibility / Best Practices / SEO)
- Vercel / Supabase の実コスト監視
- 長期運用でのメモリリーク / DB 肥大化

PR #89 で cold start 分析をしているが、定期的に再計測が必要。

### 2.5 セキュリティ深層

**自動テストがカバー**:
- CodeQL による静的解析
- pnpm audit による依存脆弱性検知
- ESLint / 独自 agent によるコード規約チェック
- 権限境界テスト (e2e で IDOR 試行等)

**人間が確認すべき**:
- ペネトレーションテスト (セッションハイジャック / CSRF / XSS bypass 等)
- 認可境界の穴 (PM/TL が admin 機能にアクセスできないか、手動で網羅)
- 監査ログの改ざん耐性 (audit_logs の WORM 性)
- 秘匿情報の実送信 (NEXTAUTH_SECRET / DATABASE_URL が log に出ていないか)
- OWASP Top 10 の人手チェックリスト

定期 (四半期) で `/threat-model` skill による STRIDE 分析を実施する運用。

### 2.6 外部サービス連携の実疎通

**自動テストがカバー**:
- Inbox provider を使った **メール送信シミュレーション** (PR #92)
- Mock された外部依存

**人間が確認すべき**:
- **実メール送信** (本番 Brevo への疎通 + 到達性 + 迷惑メール判定)
- 各メールクライアント (Gmail / Outlook / Apple Mail) でのレンダリング
- DKIM / SPF / DMARC 認証の通過
- Vercel Cron の実スケジュール動作
- Supabase 実接続 (Pooler / Direct の切替動作)

### 2.7 運用手順の実動作

**自動テストがカバー**:
- build が成功するか

**人間が確認すべき**:
- **本番 migration の実適用** (ステージング環境で本番類似データに対して)
- **ロールバック手順** (migration rollback が実際に動くか、データ損失なく戻せるか)
- バックアップとリストア (Supabase の point-in-time recovery)
- インフラ障害時の復旧手順 (Vercel 障害時 / Supabase 障害時の影響範囲)
- デプロイ手順書通りにデプロイできるか (OPERATION.md に沿って)

### 2.8 UX / 使いやすさの質的評価

**自動テストがカバー**:
- 見栄えの回帰 (視覚回帰)

**人間が確認すべき**:
- 操作感 (クリック数 / 画面遷移数の多さ / 探しやすさ)
- エラーメッセージの親切さ (「何が起きて、どうすれば良いか」が伝わるか)
- 初見ユーザが説明無しで使えるか
- ヘルプドキュメント無しで完結するか
- 多言語対応 (将来的に) 時の文字崩れ / 文脈違和感

### 2.9 データ整合性の長期運用

**自動テストがカバー**:
- 論理削除の cascade
- FK 参照整合

**人間が確認すべき**:
- 累積 10 万件規模のデータでパフォーマンス劣化しないか
- 孤児レコード発生の検知 (ProjectMember が project 削除後も残る等)
- 定期 cleanup バッチ (非アクティブ user 自動削除等) の実運用確認
- 監査ログの検索性 (実データ量で検索が実用的か)

## 3. フェーズ別 手動テスト チェックリスト

### 3.1 機能実装完了時 (PR レビュー前)

開発者本人が実施:

- [ ] 対象機能を **手順書通りに操作** して動作確認 (happy path)
- [ ] 異常系を 1-2 パターン試行 (空入力 / 巨大入力 / 連打)
- [ ] 自動テスト (ユニット + E2E + 視覚回帰) 全 green
- [ ] `pnpm lint / test / build` がローカルで通る
- [ ] CLAUDE.md コミット前チェック 5 項目を通過

### 3.2 PR マージ直前 (レビュア)

- [ ] CI 全 green を確認
- [ ] 視覚回帰 baseline 更新あり → **diff 画像をレビュー**し意図通りか確認
- [ ] カバレッジレポートで重要 path の網羅を確認
- [ ] code-reviewer agent / security-reviewer agent (必要時) の結果を確認

### 3.3 ステージング / 試験運用 (リリース候補版に昇格時)

- [ ] 主要業務フロー を **実業務担当者が** 一通り操作
- [ ] **クロスブラウザ確認** (Chrome / Firefox / Safari / Edge 各 1 回)
- [ ] **モバイル確認** (iPhone / Android で主要画面)
- [ ] 実 SMTP でメール送信 → 到達確認
- [ ] 実データ規模に近いデータで性能観察
- [ ] アクセシビリティスクリーンリーダーで主要画面を聞く

### 3.4 本番リリース前 (デプロイ直前)

- [ ] migration を **ステージングで本番類似データに適用** → 問題なし確認
- [ ] ロールバック手順を書面化 (誰が / どのコマンドで)
- [ ] 監視ダッシュボード / アラートの発火確認
- [ ] バックアップが直近取得済みであることを確認
- [ ] リリース告知準備 (ユーザ向け / 運用チーム向け)

### 3.5 定期 (月次 / 四半期)

**月次**:
- [ ] `pnpm audit` 結果の確認、critical/high 脆弱性の対応計画
- [ ] 監査ログの異常エントリ確認
- [ ] Vercel / Supabase コストの推移確認
- [ ] 累積データ量の把握 (DB サイズ / 行数)

**四半期**:
- [ ] `/threat-model` による STRIDE 脅威モデリング再実施
- [ ] ペネトレーションテスト (社内 or 外部)
- [ ] 視覚回帰 baseline の全面再生成 (意図せぬ累積ドリフト解消)
- [ ] 災害復旧訓練 (リストア手順の実地確認)
- [ ] a11y 監査 (WCAG AA 基準)

## 4. テストピラミッドと信頼性

```
                ┌──────────────┐
                │  UAT / 手動   │ ← 少量、コスト高、判断を含む
                └──────────────┘
              ┌────────────────┐
              │  視覚回帰テスト   │ ← UI 見栄え (PR #96 以降)
              └────────────────┘
          ┌──────────────────────┐
          │    E2E テスト          │ ← ユーザフロー (PR #90 以降で整備・累積)
          └──────────────────────┘
    ┌──────────────────────────────┐
    │  結合テスト / API テスト        │ ← サービス層 (既存)
    └──────────────────────────────┘
┌──────────────────────────────────────┐
│         ユニットテスト                  │ ← 最大量、コスト低、既存 646 tests
└──────────────────────────────────────┘
```

下の層ほど **高速 / 網羅的 / 低コスト**、上の層ほど **遅い / 範囲狭い / 価値集中**。

本プロジェクトは下 4 層を自動化済。**最上層の UAT / 手動テストは本文書のチェックリストで明示化** することで、「何を人間が見るべきか」が属人化せず再現可能になる。

## 5. モバイルビューポート E2E (PR #128 で追加)

### 5.1 設定

`playwright.config.ts` の `projects` に以下 2 つを定義:

```ts
projects: [
  { name: 'chromium',        use: { ...devices['Desktop Chrome'], viewport: { width: 1440, height: 900 } } },
  // chromium-mobile は iPhone 13 のエミュレーション (UA/viewport/isMobile/hasTouch) を
  // Chromium エンジンで駆動する。devices['iPhone 13'] は defaultBrowserType='webkit' を
  // 内包するため、必ず defaultBrowserType='chromium' で override する (E2E_LESSONS_LEARNED §4.35)。
  { name: 'chromium-mobile', use: { ...devices['iPhone 13'], defaultBrowserType: 'chromium' } },  // 390x844
],
```

既定で `pnpm test:e2e` は両 project を実行。モバイル視覚回帰 baseline は `*-chromium-mobile-linux.png` に別ファイルとして保存される。

### 5.2 デグレ防止戦略 (段階的 PR での運用)

各 PR #128a-d では:

| テスト種別 | デグレ検知対象 |
|---|---|
| **PC 視覚回帰 baseline (既存)** を変更しない | PC UX の退行を検知 |
| **モバイル視覚回帰 baseline を新設** | モバイル UX の退行を検知 |
| **既存の機能 E2E** をモバイル project でも実行 | 両環境で機能デグレ検知 (クリック / 入力 / ナビが両方で動く) |

### 5.3 新規 spec 追加時の注意

**PC でのみ意味がある spec** (例: Gantt 編集、並列テーブル縦整列確認) は:

```ts
test.describe('PC 専用シナリオ', () => {
  test.skip(({ browserName }, testInfo) =>
    testInfo.project.name === 'chromium-mobile', 'PC 専用');
  // ... tests ...
});
```

**モバイルでのみ意味がある spec** (例: カード化テーブルのタップ展開):

```ts
test.describe('モバイル専用シナリオ', () => {
  test.skip(({ browserName }, testInfo) =>
    testInfo.project.name === 'chromium', 'モバイル専用');
  // ... tests ...
});
```

### 5.4 視覚回帰 baseline の運用

PC + モバイルで 2 セットの baseline が必要なので、`[gen-visual]` commit で両方一括生成される。
詳細は [DEVELOPER_GUIDE.md §9.6](./DEVELOPER_GUIDE.md) 参照。

## 6. まとめ

> 自動テストは「機械が判定できること」に特化している。**判断・体感・長期・異環境** は引き続き人間の領域として残り、その運用を明示化することで品質を安定させる。

- 自動テスト green ≠ リリース可
- 自動テスト green + 本文書の該当フェーズ チェックリスト通過 = リリース可
- 人間テストで発見された問題は可能な限り **次の自動テスト** に昇格させる (LESSONS_LEARNED 追記)
