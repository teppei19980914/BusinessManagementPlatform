# プロジェクトライフサイクルとビジネスルール (Business Logic)

本ドキュメントは、プロジェクトの状態遷移・操作制限・健全性定義などの **ビジネスロジックの中核** を集約する。画面別の操作仕様は [specification/SCREENS.md](../specification/SCREENS.md)、権限マトリクスは [specification/PERMISSION_MATRIX.md](../specification/PERMISSION_MATRIX.md) を参照。

---

## システムの目的と成功条件

### 2.1 目的
本プラットフォームの目的は、プロジェクトの企画段階から完了・振り返りまでを一気通貫で支援し、プロジェクトの中止・中断を招く重大な問題を回避しながら、QCDのバランスを保って健全にプロジェクトを運営することである。

### 2.2 成功条件
本プラットフォームを通じて、プロジェクト運営に必要な情報を一元管理し、管理者およびメンバーが計画、進捗、リスク、実績、知見を継続的に共有・活用できる状態を実現する。
その結果として、顧客からの信頼獲得につながり、次案件の受注や継続的な相談機会を創出できることを成功条件とする。

---

## 健全なプロジェクト運営の定義

本プラットフォームにおける健全なプロジェクト運営とは、プロジェクトの計画、進捗、品質、コスト、リスク、意思決定、実績および知見が継続的に記録・可視化され、管理者およびメンバーが必要なタイミングで是正判断と対応を行える状態を指す。
また、プロジェクト完了後には、その運営結果と教訓が再利用可能なナレッジとして蓄積され、次案件の企画・見積もり・計画に活用できることを含む。

### 3.1 健全性を構成する観点

1. 計画の健全性
2. 進捗の健全性
3. 品質の健全性
4. コストの健全性
5. リスク・課題管理の健全性
6. 学習・再利用の健全性

### 3.2 MVPで最低限満たすべき運営条件

- プロジェクト計画を構造化して登録できる
- タスク進捗を継続的に更新できる
- 見積もりと実績を比較できる
- リスク・課題を記録し追跡できる
- 運営中に発生した判断や知見を残せる
- 完了時に振り返りを記録し再利用できる

## プロジェクト状態定義 (8 つの状態)

| 状態 | 意味 |
|---|---|
| 企画中 | 案件立上げ、目的・背景・スコープ整理を行う段階 |
| 見積中 | 見積もり作成と根拠整理を行う段階 |
| 計画中 | WBS、タスク、スケジュール、担当割当を確定する段階 |
| 実行中 | 進捗更新、実績記録、リスク/課題対応を行う段階 |
| 完了 | 作業完了後、実績整理と締め処理を行う段階 |
| 振り返り完了 | 振り返りとナレッジ化が完了した段階 |
| クローズ | 最終確定済みで、原則更新不可の段階 |

### 8.1 状態制御の原則
ユーザの操作可否は、ロール権限に加えて、プロジェクト状態によって制限される。
プロジェクト状態による制限はロール権限に優先して適用される。

## 状態遷移時のロック条件

### 10.1 企画中 → 見積中
- 目的、背景、スコープの初版が登録済みであること
- プロジェクト責任者が設定済みであること

### 10.2 見積中 → 計画中
- 見積もりが確定済みであること
- 見積根拠が登録済みであること

### 10.3 計画中 → 実行中
- WBS/タスクが作成済みであること
- 担当者、予定工数、期限が設定済みであること

### 10.4 実行中 → 完了
- 全タスクが完了またはクローズ済みであること
- 主要リスク/課題の状態が整理済みであること
- 実績工数の記録が完了していること

### 10.5 完了 → 振り返り完了
- 振り返りが記録済みであること
- 次回改善事項が登録済みであること
- ナレッジ化対象が整理済みであること

### 10.6 振り返り完了 → クローズ
- 必要なナレッジ公開が完了していること
- プロジェクトの最終状態を確定済みであること

## 権限ポリシー (2026-04-24 改修)

## 10.7 権限ポリシー (2026-04-24 改修)

リスク/課題/振り返り/ナレッジ の 4 エンティジーに共通して以下のポリシーを適用する:

1. **全○○ 画面 (/risks, /issues, /retrospectives, /knowledge)**
   - システム管理者以外: **参照のみ** (作成/編集動線なし)
   - システム管理者: **参照 + 削除のみ** (管理削除用、編集は不可)
2. **各○○一覧 画面 (/projects/[id]/risks|issues|retrospectives|knowledge)**
   - 作成: **実際の ProjectMember** (pm_tl / member) のみ
     - admin でも非メンバーのプロジェクトでは作成不可
   - 編集: **作成者本人のみ** (admin でも他人の記事は編集不可)
   - 削除: **作成者本人** (admin は全○○ から管理削除)
3. **公開種別フィルタ (visibility)**
   - 一覧画面 (全○○ / ○○一覧とも): 非 admin は `visibility='public'` のみ表示
   - admin は `visibility='draft'` も表示 (管理削除のため)
   - 個別 GET: draft は作成者本人 + admin のみ参照可、他は「存在しない」扱い (null)
4. **メモの独立方針**
   - メモは完全な個人資産。CRUD は常に自分のメモのみ、role 判定なし
   - ユーザ削除時はカスケード物理削除 (組織の資産と違い残す意味がない)

詳細は DESIGN.md §8.3 / §8.3.1 / §8.3.2 を参照。

### 13.2 アカウント登録

#### 実行者
システム管理者のみ（一般ユーザによるセルフ登録は不可）

#### 操作手順

1. システム管理者がユーザ管理画面で「新規ユーザ登録」を押下
2. 以下の項目を入力

| 項目 | 必須 | 備考 |
|---|---|---|
| ユーザ名 | ○ | 100文字以内 |
| メールアドレス | ○ | 重複不可 |
| システムロール | ○ | システム管理者 / 一般ユーザ |

※ パスワードは管理者が設定しない。ユーザ自身がパスワード設定画面で設定する。

3. 「招待メールを送信」ボタンを押下
4. 登録されたメールアドレスにパスワード設定リンクを含む招待メールが自動送信される
5. この時点ではアカウントは無効状態（ログイン不可）

#### 13.2.1 初回パスワード設定（ユーザ自身）

##### 一般ユーザ (systemRole='general')

1. ユーザが招待メール内のリンク（/setup-password）をクリック
2. パスワード設定画面でパスワード + 確認パスワードを入力（パスワードポリシー準拠）
3. 設定完了時にリカバリーコード（10個）が画面に1回限り表示される
4. ユーザはリカバリーコードを安全な場所に保管する
5. アカウントが有効化され、ログイン可能になる

##### システム管理者 (systemRole='admin') — **PR #91 で MFA 強制化**

1. ユーザが招待メール内のリンク（/setup-password）をクリック
2. **ステップ 1: パスワード設定** — パスワード + 確認パスワードを入力
   - 成功時: パスワード保存 + MFA シークレット生成 + リカバリーコード作成
   - **アカウントはまだ有効化されない** (isActive=false 維持)
   - **トークンはまだ使用済にしない** (次ステップで使う)
   - 画面は自動的に MFA 登録ステップへ遷移
3. **ステップ 2: MFA 登録** — QR コード表示 + 認証アプリで生成した 6 桁 TOTP を入力
   - 成功時のみ以下を atomic に実行:
     - `emailVerificationToken.usedAt = now`
     - `user.isActive = true`
     - `user.deletedAt = null`
     - `user.mfaEnabled = true`
     - `user.mfaEnabledAt = now`
4. **ステップ 3: リカバリーコード表示** (10 個、1 回限り)
5. ログイン画面に遷移、以後の毎回ログインで MFA 検証が必要

設計意図: 「パスワード + MFA 両方が揃わないと admin アカウントは有効化されない」
ことを技術的に強制する。中断された場合もトークンの有効期限内 (24 時間) なら
同リンクから再開可能 (パスワード設定後なら MFA ステップから)。

#### パスワードポリシー

| ルール | 要件 |
|---|---|
| 最小文字数 | 10文字以上 |
| 文字種 | 英大文字・英小文字・数字・記号のうち3種以上 |
| 最大文字数 | 128文字 |
| 禁止 | メールアドレスと同一、連続同一文字4文字以上 |
| 履歴 | 直近5回のパスワードの再利用不可 |

#### リカバリーコード

| 項目 | 仕様 |
|---|---|
| 形式 | 8文字の英数字 x 10個（例: ABCD-1234） |
| 生成 | 初回パスワード設定時に1回のみ |
| 表示 | パスワード設定完了画面で1回のみ表示。以後再表示不可 |
| 用途 | パスワードリセット時の本人確認、MFA ログイン時のバックアップ |
| 使用回数 | 各コード1回限り |

### 13.3 メール検証（アカウント有効化）

#### フロー

1. ユーザが登録時に送信されたメール内のリンクをクリック
2. システムがトークンを検証

| チェック項目 | 失敗時のメッセージ |
|---|---|
| トークンが存在するか | 「無効なリンクです」 |
| トークンが未使用か | 「既に使用されたリンクです」 |
| 有効期限内か（24時間） | 「有効期限切れです。管理者に再送を依頼してください」 |

3. 全チェック通過でアカウントが有効化される
4. ログイン画面にリダイレクト

#### 検証メール再送
- システム管理者がユーザ管理画面から実行可能
- 前回の送信から5分以上経過していること

#### 未検証アカウントの自動削除
- 7日間検証されなかったアカウントは自動で物理削除される

### 13.4 ログイン

#### 13.4.1 基本ログインフロー（MFA 未設定）

1. ログイン画面でメールアドレスとパスワードを入力
2. システムが以下を順に検証

| チェック項目 | 失敗時の動作 |
|---|---|
| ユーザが存在するか | 汎用エラーメッセージ |
| アカウントが有効か | 汎用エラーメッセージ |
| 恒久ロックされていないか | 汎用エラーメッセージ |
| 一時ロック期間中でないか | 汎用エラーメッセージ |
| パスワードが正しいか | 汎用エラーメッセージ |

※ 全ての失敗に対して同一のメッセージ「メールアドレスまたはパスワードが正しくありません」を表示（ユーザ存在有無の漏洩防止）

3. 認証成功時、セッションが発行されダッシュボードに遷移

#### 13.4.2 MFA 付きログインフロー

1. メールアドレス + パスワードの検証まで同一
2. パスワード検証成功後、MFA 入力画面に遷移（この時点ではセッション未発行）
3. 以下のいずれかで認証を完了

| 方法 | 手順 |
|---|---|
| TOTP コード | 認証アプリに表示される6桁コードを入力 |
| リカバリーコード | 未使用のリカバリーコードを1つ入力（使用後は無効化） |

- TOTP コードの試行回数: 5回まで。超過でステップ1からやり直し
- MFA 入力画面の有効期限: 5分

4. 検証成功でセッション発行、ダッシュボードに遷移

#### 13.4.3 アカウントロックポリシー

| 項目 | 条件 |
|---|---|
| 一時ロック発動 | 10分以内に5回のログイン失敗 |
| 一時ロック期間 | 30分間 |
| 恒久ロック発動 | 一時ロックが3回発生 |
| 恒久ロック解除 | システム管理者が手動で解除 |

#### 13.4.4 ログインエラーメッセージ (PR #87 で差別化)

ログイン失敗時、`POST /api/auth/lock-status` で対象メールのロック状態を参照し、
画面上のエラー文言を 3 パターンで切り替える。ロック中ユーザに「パスワード誤り」と
表示する旧挙動では、ユーザが「何度試してもログインできない」状態に気付けず
問い合わせが増加するため。

| 状況 | 表示メッセージ |
|---|---|
| 永続ロック | アカウントがロックされています。管理者に解除を依頼してください。 |
| 一時ロック (期限内) | ログイン失敗が続いたためアカウントが一時ロックされています。YYYY-MM-DD HH:MM 以降に再度お試しください。 |
| それ以外 (パスワード誤り / 未存在メール 等) | メールアドレスまたはパスワードが正しくありません |

#### 13.4.5 ユーザ削除とカスケード (PR #89 で実装、feat/account-lock で 30 日経過時の挙動を変更)

| 項目 | 仕様 |
|---|---|
| 手動削除トリガ | 管理画面「このユーザを削除」ボタン |
| 手動削除方式 | User 本体: **論理削除** (deletedAt セット + isActive=false + MFA 無効化) |
| 手動削除カスケード | ProjectMember / Session / RecoveryCode / EmailVerificationToken / PasswordResetToken / PasswordHistory を**物理削除** |
| **自動ロックトリガ** (feat/account-lock 改修) | 日次 cron による 30 日無アクティブの自動 **ロック** (旧仕様: 自動削除) |
| 自動ロック条件 | `lastLoginAt` (未ログインの場合 `createdAt`) から 30 日経過 + systemRole='admin' 以外 |
| **自動ロック方式** | `isActive=false` のみ更新 (deletedAt 未セット、ProjectMember も維持) |
| 自動ロック解除 | `/admin/users` で当該ユーザを編集 → 有効化トグルを ON → 保存 |
| 設計意図 | 過去ナレッジ等の **作成者表示** はアカウントが残っていないと「(削除済)」になるため、長期不在ユーザでもアカウント情報を保持する。一方、攻撃面 (漏洩パスワード / 放置セッション) を縮小するため **ログインだけは封じる** |
| 保全対象 | Task.assigneeId / RiskIssue.reporterId / Knowledge.createdBy 等の scalar 参照は残す (履歴保全) |
| 自己削除 | 禁止 (CANNOT_DELETE_SELF) |
| cron 認証 | Vercel Cron + `Authorization: Bearer <CRON_SECRET>` (環境変数) |

**関連**:
- `vercel.json` の `crons` で `/api/admin/users/lock-inactive` を日次 03:00 UTC 実行 (旧名 `cleanup-inactive`、feat/account-lock で改名)
- `src/config/security.ts` の `INACTIVE_USER_LOCK_DAYS` (既定 30、旧名 `INACTIVE_USER_DELETION_DAYS`) で閾値変更可
- 監査ログ: `audit_log` に action='UPDATE' / entityType='user' / after.reason='30 日無アクティブ自動ロック' で記録

#### 13.4.6 プロジェクト削除カスケードの細粒度制御 (PR #89)

プロジェクト削除ダイアログで「どの資産を削除するか」を 4 つのチェックボックスで個別指定:

| チェック項目 | 対象エンティティ |
|---|---|
| リスク一覧 | `riskIssue` WHERE type='risk' + 紐づく `attachment` |
| 課題一覧 | `riskIssue` WHERE type='issue' + 紐づく `attachment` |
| 振り返り一覧 | `retrospective` + `retrospectiveComment` + 紐づく `attachment` |
| ナレッジ一覧 | `knowledge` (単独紐付けのみ物理削除、共有は紐付け解除のみ) + 紐づく `attachment` |

チェックを入れなかった項目は物理削除されず、全○○ 画面で「親プロジェクト削除済みの孤児資産」として残る。

**強制削除対象 (チェック不能、常に物理削除)**:
プロジェクト本体 / Task + TaskProgressLog (WBS・ガント) / Estimate (見積) / ProjectMember / 上記に紐づく Attachment 全て。

#### 13.4.7 Attachment カスケード削除 (PR #89)

親エンティティ (project / task / estimate / risk / retrospective / knowledge / memo) を削除したとき、
紐づく `attachment` も同時に論理削除する。UI からアクセスできない孤児 attachment が DB に残留するのを防ぐ。

| 親エンティティ | 削除サービス | attachment の削除タイミング |
|---|---|---|
| Task | `deleteTask` | $transaction 内で同時 |
| Estimate | `deleteEstimate` | $transaction 内で同時 |
| Risk/Issue | `deleteRisk` | $transaction 内で同時 |
| Retrospective | `deleteRetrospective` | $transaction 内で同時 |
| Knowledge | `deleteKnowledge` | $transaction 内で同時 |
| Memo | `deleteMemo` | $transaction 内で同時 |
| Project (論理削除) | `deleteProject` | project / task / estimate 配下を同時 |
| Project (物理カスケード) | `deleteProjectCascade` | 削除する全エンティティの attachment を物理削除 |

**enumeration 対策**: `lock-status` API は存在しないメールでも `status: 'none'` を返し、
ロックされていない既存ユーザと区別できないようにしている。ロック状態の開示は、
既に 5 回失敗で自動ロックする挙動自体が観測可能であるため新規の情報漏洩リスクとはならない。

### 13.8 MFA（多要素認証）設定

#### 13.8.1 MFA 有効化フロー

**一般ユーザ (任意 / 後付け)**:
1. 設定画面で「多要素認証を有効化」を選択
2. QR コードが表示される
3. 認証アプリ（Google Authenticator 等）でスキャンし登録
4. 確認のため TOTP コード（6桁）を1回入力
5. 検証成功で MFA が有効化される

**システム管理者 (必須 / 初期セットアップで強制)** — PR #91 で変更:
- §13.2.1 の初期セットアップフローで **パスワード設定と同時に MFA 登録が必須**
- 設定画面からの手動有効化は不要 (初期セットアップで必ず有効化される)

| 項目 | 仕様 |
|---|---|
| 方式 | TOTP（6桁、30秒間隔） |
| 管理者 | **MFA 強制有効化 (PR #91)** — 初期セットアップで必須、設定画面から解除不可 |
| 一般ユーザ | 任意設定 (従来通り) |

#### 13.8.2 MFA 無効化フロー

**一般ユーザ**:
1. 設定画面で「多要素認証を無効化」を選択
2. MFA が無効化される (TOTP 再確認は現状なし)

**システム管理者**: ❌ **無効化不可** (PR #91)
- 設定画面にボタン自体が表示されない (UI ガード)
- API `POST /api/auth/mfa/disable` が 403 を返す (route ガード)
- サービス層 `disableMfa()` が `CANNOT_DISABLE_ADMIN_MFA` を throw (サービスガード)
- → 3 層の防御で admin が MFA 解除する経路を完全遮断

#### 13.8.3 MFA verify ブルートフォース防御 (PR #116 / 2026-04-24)

パスワード認証通過後の MFA TOTP 入力にもロック機構を組み込む。

| 項目 | 仕様 |
|---|---|
| 閾値 | **3 回** 連続失敗 (パスワードロックの 5 回より厳しめ。TOTP 空間の総当たり耐性を強化) |
| ロック期間 | **30 分** (パスワードロックと同じ) |
| 永続ロック | **設けない** — recovery code による自己解除が可能なため、「admin 介入なしで永遠にログイン不可」状態を生む必要がない |
| DB 列 | `users.mfa_failed_count` / `users.mfa_locked_until` (PR #116 追加、パスワード系 `failed_login_count` / `locked_until` とは別系統) |
| HTTP 応答 | ロック中の `/api/auth/mfa/verify` は **429 Too Many Requests** + `{error.code: 'MFA_LOCKED', error.lockedUntil: <ISO8601>}` |
| 監査ログ | `auth_event_logs` に `eventType='lock', detail.lockType='mfa_temporary'` を記録 |

**解除経路**:

| 経路 | 条件 | 効果 |
|---|---|---|
| 時間経過 | `mfa_locked_until < now()` | 自動解除、次回 verify でカウンタも 0 にリセット |
| リカバリーコード入力 | `/api/auth/mfa/verify` に `body.recoveryCode` を送って 1 回だけ有効 | 即座にロック解除 + `mfa_failed_count = 0` |
| admin 手動解除 | `/api/admin/users/[id]/unlock` (既存) | パスワードロックと同時に MFA ロックもリセット |

**カウンタの挙動**:
- 正解 TOTP / recovery code 入力 → `mfa_failed_count = 0` にリセット
- 失敗 → increment
- 閾値到達 (3 回目の失敗) → `mfa_locked_until = now + 30min` セット + `mfa_failed_count = 0` にリセット (次ロックサイクル準備)

**admin 画面表示** (/admin/users):
- パスワードロック / MFA ロック / 永続ロック / 失敗カウント を 1 列「認証ロック」に集約
- tooltip で原因・解除予定・解除手段を表示 (A 案採用、DESIGN §9.4 参照)

### 13.13 未使用アカウントの自動削除

#### 段階的な処理

| 段階 | タイミング | 処理 |
|---|---|---|
| 警告メール | 最終ログインから23日後 | 「7日以内にログインしないとアカウントが無効化されます」 |
| 論理削除 | 最終ログインから30日後 | アカウント無効化 + 全セッション無効化 + 通知メール |

#### 除外条件
- システム管理者ロールのユーザは自動削除の対象外

#### 論理削除中の復帰
- ユーザがログインすればアカウントが即座に復活する
- プロジェクトメンバーシップは復元されない（管理者が再追加）

#### 物理削除時のデータの扱い

| データ種別 | 処理 |
|---|---|
| 個人情報（氏名・メール） | 完全削除 |
| パスワード・リカバリーコード・MFA 情報 | 完全削除 |
| ユーザが作成したナレッジ・タスク等 | 「削除済みユーザ」名義で保持 |
| 監査ログ・操作ログ | 匿名化して保持（2年間） |
