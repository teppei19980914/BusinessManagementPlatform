# たすきば Knowledge Relay 要件定義書（たたき台）

## 1. 文書概要

### 1.1 文書の目的
本書は、プロジェクトの企画段階から完了・振り返りまでを一気通貫で支援するプラットフォームについて、現時点で整理できている要件をまとめたたき台である。  
本書をもとに、今後、業務フロー、画面仕様、データ項目、権限、非機能要件などを詳細化していく。

### 1.2 背景
各プロジェクトで発生するリスク、調査・検証結果、意思決定、実績、教訓などの知見は、プロジェクト運営や次案件への再利用において重要である。  
一方で、知見が分散管理されると、見積もり、WBS作成、進捗管理、リスク対応、振り返りに活かしづらくなる。  
そのため、本プラットフォームでは、ナレッジを中核に据えつつ、プロジェクトの最初から最後まで継続利用できる一気通貫の運営基盤を実現する。

---

## 2. システム概要

### 2.1 システムの位置づけ
本システムは単一機能アプリではなく、複数機能を有するプロジェクト運営プラットフォームである。  
プロジェクトの企画、見積もり、計画、実行、監視、完了、振り返りまでを一つの基盤上で扱い、継続的に利用することを前提とする。

### 2.2 基本コンセプト
本プラットフォームは、以下の流れを切れ目なく支援することを目指す。

- 企画
- 見積もり
- 計画立案
- 実行管理
- リスク／課題対応
- 実績蓄積
- 振り返り／ナレッジ化
- 次案件への再利用

### 2.3 中核となる考え方
本プラットフォームの中核は、各プロジェクトで得られた知見を一元管理し、それを将来の意思決定や判断支援に再利用できる状態を作ることである。  
MVPでは、AIによる高度な自動化よりも前に、情報を構造化して蓄積し、実運用で継続更新できる基盤を整えることを優先する。

---

## 3. システムの目的と成功条件

### 3.1 目的
本プラットフォームの目的は、プロジェクトの企画段階から完了・振り返りまでを一気通貫で支援し、プロジェクトの中止・中断を招く重大な問題を回避しながら、QCDのバランスを保って健全にプロジェクトを運営することである。

### 3.2 成功条件
本プラットフォームを通じて、プロジェクト運営に必要な情報を一元管理し、管理者およびメンバーが計画、進捗、リスク、実績、知見を継続的に共有・活用できる状態を実現する。  
その結果として、顧客からの信頼獲得につながり、次案件の受注や継続的な相談機会を創出できることを成功条件とする。

---

## 4. 健全なプロジェクト運営の定義

### 4.1 定義
本プラットフォームにおける健全なプロジェクト運営とは、プロジェクトの計画、進捗、品質、コスト、リスク、意思決定、実績および知見が継続的に記録・可視化され、管理者およびメンバーが必要なタイミングで是正判断と対応を行える状態を指す。  
また、プロジェクト完了後には、その運営結果と教訓が再利用可能なナレッジとして蓄積され、次案件の企画・見積もり・計画に活用できることを含む。

### 4.2 健全性を構成する観点

#### 4.2.1 計画の健全性
- プロジェクトの目的、スコープ、WBS、スケジュール、担当が明確であること
- 見積もり根拠が残っていること
- マイルストーンが定義されていること

#### 4.2.2 進捗の健全性
- タスク進捗が継続的に更新されること
- 遅延や滞留が早期に把握できること
- 実績工数が把握できること

#### 4.2.3 品質の健全性
- 品質上の懸念や問題が記録されること
- 問題と対応内容が紐づいていること
- 再発防止の知見が残ること

#### 4.2.4 コストの健全性
- 見積工数と実績工数を比較できること
- 差分理由を説明できること
- コスト逸脱の兆候を把握できること

#### 4.2.5 リスク・課題管理の健全性
- リスクが事前に起票されること
- 優先度、影響度、担当、対応策が整理されていること
- 課題が放置されず、追跡可能であること

#### 4.2.6 学習・再利用の健全性
- 振り返りが記録されること
- 教訓がナレッジ化されること
- 次案件の企画・見積もり・計画に再利用されること

## 5. 想定ユーザーと利用形態

### 5.1 想定ユーザー
本プラットフォームでは、主に以下の利用者を想定する。

#### 5.1.1 プロジェクトマネージャ／チームリーダ
- 過去知見や実績を参照する
- 見積もりを作成する
- WBSを作成する
- ガントチャートを作成する
- メンバーへタスクを割り当てる
- 進捗、遅延、リスク、課題を把握する
- プロジェクト終了時の振り返りとナレッジ化を行う

#### 5.1.2 メンバー
- 自分の担当タスクを確認する
- 進捗率、ステータス、実績工数を更新する
- コメントや作業メモを残す
- 必要に応じて知見を登録する

#### 5.1.3 管理者
- 利用者、権限、各種設定を管理する
- プラットフォーム全体の運用ルールを維持する

### 5.2 利用前提
本プラットフォームは、プロジェクトの企画段階から終了後の振り返りまで継続利用することを前提とする。  
そのため、管理者だけでなくメンバーもアクセスし、プロジェクト実行中の情報更新を行える必要がある。

## 6. MVPの考え方

### 6.1 基本方針
MVPでは、すべての機能を高度に作り込むのではなく、プロジェクトライフサイクル全体を切れ目なくつなぐ最小構成を実現する。  
特に、以下の業務フローが一つのプラットフォーム上で成立することを重視する。

1. 企画から見積もりまで  
2. 見積もりから計画作成まで  
3. 実行から監視まで  
4. 完了から次案件への再利用まで

### 6.2 MVPで最低限実現すべき状態
- 管理者が過去の知見や実績を参照しながら、見積もりと計画を立てられる
- 管理者がWBSとガントを作成し、メンバーにタスクを展開できる
- メンバーが同じアプリ上で進捗、実績、コメントを更新できる
- 実行中に発生したリスク、課題、判断、学びを記録できる
- プロジェクト完了時に実績と教訓を振り返り、次案件へ再利用できる

## 7. MVP必須機能

### 7.1 プロジェクト立ち上げ機能
- プロジェクト作成
- 基本情報登録（目的、背景、開発方式、期間、体制など）
- ステータス管理
- メンバー紐付け
- 類似案件／参考ナレッジ紐付け

### 7.2 ナレッジ登録・参照・検索機能
- ナレッジ作成、編集、閲覧
- 種別分類（調査、検証、障害、意思決定、教訓、ベストプラクティスなど）
- タグ付け
- プロジェクト、タスク、リスク、見積もりとの紐付け
- キーワード、タグ、開発方式、工程などによる検索

### 7.3 見積もり管理機能
- プロジェクト単位の見積もり作成
- 作業項目ごとの見積工数登録
- 見積根拠記録
- 過去ナレッジ／過去実績の参照リンク紐付け
- 実績との比較

### 7.4 WBS／タスク管理機能
- WBSの階層管理
- タスク作成
- 担当者、開始日、終了日、予定工数の設定
- マイルストーン管理
- ステータス管理
- 進捗率管理
- **CSV エクスポート / 上書きインポート** (feat/wbs-overwrite-import で追加)
  - エクスポート: 画面表示中の全列 + DB 上の ID + 進捗系参考情報を含む CSV を出力
  - 上書きインポート: ID 列で既存タスクと突合し、ID 一致=UPDATE / 空欄=新規作成
    で同一プロジェクト内の WBS を Excel 編集サイクルで再インポート可能にする
  - 進捗・実績データは保全 (CSV 上では read-only として表示するが import 時は無視)
  - 実行前に dry-run プレビューで差分を確認できる必須ステップを設ける
  - 削除候補のうち進捗を持つタスクは赤エラーで強調表示し、誤削除を予防する
  - インポート途中でエラー発生時はインポート前の状態に完全ロールバック
  - 認可: PM/TL + admin のみ実行可 (member 以下は実行不可)

### 7.5 ガントチャート表示機能
- タスクのガント形式表示
- 期間、担当者、進捗率の表示
- マイルストーン表示
- 遅延タスクの識別
- 全体表示および担当者別表示

### 7.6 メンバー向け進捗更新機能
- 自分の担当タスク一覧表示
- 進捗率更新
- ステータス更新
- 実績工数入力
- コメント／作業メモ入力
- 完了報告

### 7.7 リスク・課題管理機能
- リスク登録
- 課題登録
- 重要度、影響度、優先度設定
- 状態更新
- 対応策記録
- プロジェクト、タスク、ナレッジとの紐付け

### 7.8 実績管理機能
- タスク単位または作業項目単位での実績工数保持
- 見積工数との比較
- 差分理由登録
- プロジェクト単位での計画対実績確認

### 7.9 振り返り・ナレッジ化機能
- プロジェクト終了時の振り返り記録
- 良かった点、問題点、差分要因、改善事項の記録
- 実行中に蓄積した情報を参照しながらの整理
- 再利用可能なナレッジとしての公開

### 7.10 権限・ロール管理機能
- 管理者、PM/TL、メンバーのロール設定
- プロジェクト単位のメンバー割り当て
- 閲覧・編集可能範囲の制御

### 7.11 ステークホルダー管理機能 (PMBOK 13 / feat/stakeholder-management で追加)
- プロジェクトに関与する全関係者 (内部メンバー + 外部関係者) を 1 テーブルで管理
- Mendelow Power/Interest grid (影響度 1-5 × 関心度 1-5) による分類
  - 影響大 × 関心大 (最重要): 密接に連携 (Manage Closely)
  - 影響大 × 関心小 (要警戒): 満足させておく (Keep Satisfied)
  - 影響小 × 関心大 (情報源): 常に情報を伝える (Keep Informed)
  - 影響小 × 関心小 (監視): モニタリング (Monitor)
- 姿勢 (Attitude) 記録: 賛成 / 中立 / 反対
- エンゲージメント Gap (PMBOK 13.1.2): 現在のエンゲージメント vs 望ましいエンゲージメント
  (unaware → resistant → neutral → supportive → leading の 5 段階) を保持し、Gap が 0 で
  ない関係者を「能動的働きかけ必要」として可視化
- 人となり / 考え方の自由記述 + 検索用タグ
- 対応戦略の具体的アクション記録
- 可視性: 個人情報・人物評を含むため **PM/TL + admin のみ閲覧可** (member 以下は不可)

## 8. MVP対象外（現時点）

MVPでは、以下の機能は対象外とする。

- AIによる自動見積もり
- AIによるWBS自動生成
- 類似案件自動推薦
- リスク予兆検知
- 高度なガント編集（ドラッグ操作等）
- 高度な通知制御
- 承認ワークフロー
- チャット機能
- 高度BIダッシュボード
- 外部サービスとの複雑な双方向連携

## 12. 要件定義の暫定結論

本プラットフォームのMVPは、単なるナレッジ管理ツールではなく、プロジェクトの企画、見積もり、計画、実行、監視、完了、振り返りまでを一つの基盤上で連続的に扱うためのプロジェクト運営プラットフォームである。  
管理者とメンバーが同じ基盤上で計画、進捗、リスク、実績、知見を継続的に更新・活用し、プロジェクトを健全に完遂し、その結果を次案件へ再利用できる状態をMVPの成立条件とする。

## 13. 提案エンジン強化要件 (T-03、2026-06-01 リリース)

本セクションは、本サービスの核心機能であり最大の差別化ポイントとなる「提案エンジン」の根本的な性能向上に関する要件を定義する。本機能は外部 LLM API への継続的な金銭コストが発生する初の機能であり、悪用された場合の経済的損失リスクが極めて高い。そのため機能要件と非機能要件の両面で、特にセキュリティと経済的安全性に重点を置いた要件定義を行う。

### 13.1 背景と目的

現状の提案エンジン (PR #65) は文字 n-gram 類似度とユーザ手動入力タグの Jaccard 係数で動作しており、文章の意味的な近さを捉えられない構造的弱点を抱える。本要件で定義する強化により、ユーザは過去の知見を意味レベルで自動的に発見でき、用語の揺れやシノニムを意識せず、自分のプロジェクト文脈に関連する過去事例・ナレッジ・振り返りを得られるようになる。これは「ナレッジは溜まるが利活用されない」という既存 PM ツール共通の課題に対する本サービスの差別化解答である。

### 13.2 機能要件

**F-13.1 LLM による自動タグ抽出**: ユーザがプロジェクトの purpose / background / scope を入力した時点で、システムは LLM を用いてこれらの文章を解析し、businessDomainTags / techStackTags / processTags にあたるタグを自動的に抽出して保存しなければならない。ユーザが手動でタグを編集した場合は、その編集を保持し、以降の自動抽出は手動入力を上書きしないものとする。

**F-13.2 意味類似度に基づく提案**: システムは過去のナレッジ・解消済み課題・振り返りに対して、ユーザの新規プロジェクトの purpose / background / scope と意味的に類似する候補を embedding ベクトルの Cosine Similarity で評価し、関連度順に提示しなければならない。用語の表面的な一致 (例: 同じ単語の出現) ではなく、概念的な近さ (例: 「ログイン認証エラー」と「サインインの不具合」) を発見できることを必須とする。

**F-13.3 3 軸統合スコアリング**: 提案候補のスコアは、タグ Jaccard 係数・pg_trgm 文字一致・embedding 意味類似の 3 軸の重み付き合成で算出されなければならない。各重みは設定値として外出しし、運用中の調整を可能とする。embedding 重みを最も高く設定し、意味類似を主軸に据える。

**F-13.4 初期シードデータの提供**: 新規ユーザでも提案エンジンが価値を発揮できるよう、システムは初期データとして「IT 系資格試験 (IPA / PMP / CompTIA 等) で扱われる事例」「業界に広く知られた法則 (Brooks の法則、Conway の法則等)」を独自に要約したナレッジを 30〜100 件、システム所有プロジェクトに事前登録しておかなければならない。著作権上の安全性のため、これらは公開事例を参照しつつ要約は独自に書き起こしたものとする。

**F-13.5 プラン別動作**: システムは User の subscription_tier (free / pro / pro_trial) を参照し、プランに応じて LLM のモデルを切り替えなければならない。free プランは Claude Haiku を使用し、Pro プランは将来的に Claude Sonnet を使用する (Pro 機能の Sonnet 切替はバージョンアップで実装、6月1日リリースには含まない)。

### 13.3 非機能要件

**NF-13.1 経済的安全性**: システムは、悪意あるユーザまたはバグによる LLM API 呼び出しの暴走から運用者の金銭的損害を防ぐため、3 重の rate limit 機構を実装しなければならない。第一に Upstash Redis を用いた API ルート単位の短期 rate limit (1 ユーザ / 1 分 / 10 回、1 ユーザ / 1 時間 / 60 回)、第二にアプリケーション層でのユーザ単位月間トークン上限 (free: 10 万、pro: 100 万)、第三に Anthropic workspace 全体の月間ハード上限 (想定使用量の 1.5〜2 倍) を多層的に設定する。これら 3 層のいずれかが破られても、最後の workspace 上限で必ず止まることを担保する。

**NF-13.2 シークレット保護**: API キー (Anthropic / Voyage AI) は Vercel 環境変数のみに格納し、コード・コミット履歴・ログのいずれにも絶対に含めてはならない。git pre-commit hook で gitleaks による検知を実装し、GitHub Push Protection を有効化することで二重防御を行う。

**NF-13.3 プロンプトインジェクション耐性**: ユーザ入力に含まれる悪意あるプロンプト (システム指示の上書きを試みる文字列等) によって LLM が意図しない出力をすることを防がなければならない。具体的には入力長の厳格な制限、システムプロンプトとユーザ入力の XML タグでの明確な分離、LLM 出力の zod スキーマによる構造化検証、コンテキスト隔離 (他ユーザの個人情報や admin 情報をプロンプトに含めない) を実装する。

**NF-13.4 監視と異常検知**: システムは LLM の呼び出しごとに userId・トークン数・レスポンス時間・IP・User-Agent を記録し、日次バッチで集計して異常 (通常の 3 倍を超えるコスト発生・特定ユーザの異常使用) を検知した場合に admin に通知しなければならない。これは将来的にサービス内のダッシュボード `/admin/observability/llm` として可視化される予定であり、その前提となるデータモデルを v1 から整備する。

**NF-13.5 認可境界の遵守**: 提案エンジンが LLM や embedding 検索に渡すデータには、絶対に他ユーザの draft (visibility=private) データや admin 情報を含めてはならない。検索クエリの WHERE 句で必ず visibility='public' を指定し、type-safe に enforce する。

**NF-13.6 障害時の優雅な縮退**: LLM API (Anthropic) や Embedding API (Voyage AI) の障害時には、システムは embedding ベースの並びだけを返す、または既存の pg_trgm + Jaccard ベースの旧スコアにフォールバックすることで、全体の提案機能が停止しないように設計しなければならない。

**NF-13.7 不正アカウント乱造の抑止**: 大量の捨てアドレスによる無料枠悪用を防ぐため、サインアップ時に Cloudflare Turnstile (または同等の bot 対策) を導入し、IP 単位のサインアップ rate limit (1 IP / 1 時間 / 5 件) を設定しなければならない。

### 13.4 ライセンスとオープンソース戦略

本機能を含むコードベース全体は AGPL ライセンスで GitHub に公開する。これにより、コードはオープンに、運用 (ホスティング) はクラウド SaaS として有料提供する「オープンコア (Open Core) モデル」を採る。AGPL の採用により、競合他社が本コードベースを用いてクローズドな商用 SaaS を立ち上げることを実質的に阻止し、本サービスの事業価値を保護する。

### 13.5 段階的リリース計画

提案エンジンの全機能は 6月1日リリースで一括投入するのではなく、段階的にリリースする。**v1 (6月1日)** には Phase 1 (LLM 自動タグ抽出)、Phase 2 (pgvector + Embedding 意味検索)、初期シードデータ、subscription_tier 基盤、5 層悪用防止、監視・異常検知の最小実装を含める。**v1.x (6月中旬以降)** で Phase 3 (LLM Re-ranking と説明文生成)、Sonnet ティーザー機能、30 日無料試用機能、観測ダッシュボード UI を順次追加する。

詳細な実装計画は [SUGGESTION_ENGINE_PLAN.md](./SUGGESTION_ENGINE_PLAN.md) を、機能仕様は [SPECIFICATION.md §26](./SPECIFICATION.md) を、設計は [DESIGN.md §34](./DESIGN.md) を、脅威モデルは [docs/security/SUGGESTION_ENGINE_THREAT_MODEL.md](../security/SUGGESTION_ENGINE_THREAT_MODEL.md) を参照する。

### 13.6 マルチテナント運用と外部公開後の動線 (T-03 補強要件)

提案エンジン v2 の設計と同時に、本サービスは外部公開後にマルチテナント SaaS として運用される前提で要件を整備する。これは本機能の経済的安全性 (悪用された場合の損失をテナント単位に閉じ込める) と、外部ユーザの心理的安全性 (運用者および他テナントから自分のデータが見えない) の両立を実現するために必須となる。設計詳細は [DESIGN.md §34.11](./DESIGN.md) を参照。

#### 13.6.1 外部公開後の運用フローに関する要件

**F-13.6 テナント分離**: システムはテナントを業務データの最上位の認可境界とし、テナント間でデータが完全に分離されなければならない。テナント A のユーザはテナント B のデータを閲覧・更新・削除できず、運用者 (admin) も他テナントの内部データにアクセスできない (相互不可視性)。これは admin の信頼性に依存しない構造的な分離 (アプリケーション層で `tenantId` フィルタを必須化) で実装する。

**F-13.7 テナント招待と初期データ投入**: 外部ユーザの利用申し込みを受けて運用者が新テナントを作成した時点で、システムは当該テナントに **初期シードデータ (資格試験事例・著名な法則の独自要約) を自動的に投入** しなければならない。これは新規ユーザがサインアップ直後から提案エンジンの価値を体験できるようにするための必須機能で、初期離脱を防ぐ役割を果たす。

**F-13.8 サブスク契約のテナント単位適用**: ユーザが Pro プランを契約した場合、その契約は当該ユーザではなく **当該テナント全体** に適用される。テナント内のすべてのユーザが Pro 機能を享受でき、契約解除時はテナント全体が Free プランに戻る。これは個人利用と組織利用の両方を統一的に扱える課金モデルとなる。

**F-13.9 利用停止時のテナント削除**: 外部ユーザが利用停止を申し出た場合、システムは当該テナントとそれに紐付く全業務データを物理削除しなければならない。これは生成 AI の悪用 (退会後の API 連打など) を防ぐ目的と、ユーザのデータ所有権 (削除権) を尊重する目的の両方を兼ねる。法的な保存義務がある特定のログ (監査ログ等) は別途匿名化して保管する例外を設ける。

#### 13.6.2 テナント単位のコスト管理に関する非機能要件

**NF-13.8 テナント単位の利用上限**: §13.3 の NF-13.1 で定めた月間トークン上限は、ユーザ単位ではなく **テナント単位** で管理する。テナント内の複数ユーザが同じ予算を共有し、契約 (=テナント) と予算管理が完全に一致する。Free テナントは月 100,000 トークン、Pro テナントは月 1,000,000 トークンを上限とする。

**NF-13.9 提案機能の多用に対するコスト保護**: ユーザが「参考」タブなど提案機能を多用することによるコスト暴発を防ぐため、システムは以下の 3 段階のコスト保護を実装しなければならない。第一に、Phase 3 の LLM Re-ranking 結果を `(tenantId, projectId, contentHash)` キーで 5〜10 分間キャッシュし、繰り返し表示時の LLM 呼び出しを抑制する。第二に、テナント単位の **日次 LLM 呼び出しキャップ** (Free 30 回 / Pro 200 回) を設定し、超過時は Phase 3 をスキップして Phase 2 (embedding ベースの並びのみ) を返す。第三に、月間トークン上限と Anthropic workspace 上限が最終防衛線として機能する。

**NF-13.10 テナント認可境界の徹底**: すべての API ルートと Server Action は、操作対象データの `tenantId` がリクエストユーザの `tenantId` と一致することを最初の行で検証しなければならない。`@/lib/permissions.ts` に `requireSameTenant(user, entity)` ユーティリティを実装し、認可ロジックの入り口として機能させる。テナント境界の越境を許容する経路は absolute admin 操作 (運用者がテナントを管理する画面) のみとし、これも厳格に admin role を要求する。

#### 13.6.3 インフラスケーラビリティに関する要件

**NF-13.11 インフラ移行可能性の維持**: システムは特定インフラ (Vercel / Supabase) に強く依存せず、将来的な AWS / Azure / GCP への移行を 1〜2 週間程度の工数で実現できる構造を維持しなければならない。Prisma による DB プロバイダ抽象化、Next.js の `output: 'standalone'` 対応、Docker 化可能な構成、を維持する。Vercel 固有の API (例: `@vercel/cron`) を使う場合は、移行時の置換ポイントを明確にコード内コメントで記録する。

**NF-13.12 移行判断トリガーの明文化**: インフラ移行を判断する具体的なトリガー条件 ([DESIGN.md §34.13.3](./DESIGN.md) 参照) を運用ドキュメントに明記し、定期的に状況を評価する仕組みを設ける。早期過剰投資と判断遅延の両方を避ける。

#### 13.6.4 v1 と v1.x の段階的実装

v1 (6月1日) では、**「単一デフォルトテナント」での運用** に絞ってマルチテナント基盤を実装する。Tenant テーブルの新設、全業務エンティティへの `tenantId` カラム追加、既存データの default-tenant への migration、テナント境界を強制する認可ロジックの整備、を v1 の必達範囲とする。実質的には 1 テナントのみが存在する状態で稼働するが、コード上はマルチテナント完全対応となる。

v1.x では、テナント管理 UI (admin 専用)、新規テナント作成時のシードデータ自動投入、テナント削除時のカスケード削除、招待メール、Stripe 連携、テナント slug の URL ルーティング (path-based)、を順次追加する。これらは外部ユーザの本格的な受け入れと並行して実装する。
