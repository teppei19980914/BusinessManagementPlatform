## 1. 文書概要

### 1.1 目的
本仕様書は、プロジェクトの企画段階から完了・振り返りまでを一気通貫で支援する「プロジェクト運営プラットフォーム」のMVP仕様を定義することを目的とする。

### 1.2 対象
本仕様書は、以下を対象とする。

- MVPの対象範囲
- 利用ロールと権限
- プロジェクト状態遷移
- 状態ごとの操作制限
- 主要画面仕様
- 一覧項目・入力項目・ボタン制御

### 1.3 前提
本プラットフォームは、単一機能の業務アプリではなく、プロジェクトの企画から終了後の振り返り・知見再利用まで継続利用する一気通貫型プラットフォームとして利用する。

---

## 2. システムの目的と成功条件

### 2.1 目的
本プラットフォームの目的は、プロジェクトの企画段階から完了・振り返りまでを一気通貫で支援し、プロジェクトの中止・中断を招く重大な問題を回避しながら、QCDのバランスを保って健全にプロジェクトを運営することである。

### 2.2 成功条件
本プラットフォームを通じて、プロジェクト運営に必要な情報を一元管理し、管理者およびメンバーが計画、進捗、リスク、実績、知見を継続的に共有・活用できる状態を実現する。
その結果として、顧客からの信頼獲得につながり、次案件の受注や継続的な相談機会を創出できることを成功条件とする。

---

## 3. 健全なプロジェクト運営の定義

本プラットフォームにおける健全なプロジェクト運営とは、プロジェクトの計画、進捗、品質、コスト、リスク、意思決定、実績および知見が継続的に記録・可視化され、管理者およびメンバーが必要なタイミングで是正判断と対応を行える状態を指す。
また、プロジェクト完了後には、その運営結果と教訓が再利用可能なナレッジとして蓄積され、次案件の企画・見積もり・計画に活用できることを含む。

### 3.1 健全性を構成する観点

1. 計画の健全性
2. 進捗の健全性
3. 品質の健全性
4. コストの健全性
5. リスク・課題管理の健全性
6. 学習・再利用の健全性

### 3.2 MVPで最低限満たすべき運営条件

- プロジェクト計画を構造化して登録できる
- タスク進捗を継続的に更新できる
- 見積もりと実績を比較できる
- リスク・課題を記録し追跡できる
- 運営中に発生した判断や知見を残せる
- 完了時に振り返りを記録し再利用できる

## 4. MVP対象範囲

### 4.1 ライフサイクル対象
MVPでは以下のフェーズを対象とする。

1. 企画・立ち上げ
2. 見積もり・計画
3. 実行
4. 監視・調整
5. 完了・実績確定
6. 振り返り・ナレッジ化

### 4.2 MVP必須機能

- プロジェクト立ち上げ
- ナレッジ蓄積・検索
- 見積もり管理
- WBS / タスク管理

- リスク・課題管理
- 実績管理
- 振り返り・ナレッジ化
- 権限管理

### 4.3 MVP対象外

- AI自動見積もり
- AI自動WBS生成
- ガント上の直接編集
- 高度な通知条件設定
- 高度BIダッシュボード
- 複雑な外部サービス双方向連携
- チャット/掲示板機能
- 承認ワークフロー

## 6. 利用ロールと権限制御方針

### 6.1 権限制御方式
本システムでは、ロールベースアクセス制御（RBAC）を採用する。

ユーザは以下の2種類のロールを持つ。

- **システムロール**: ユーザに対して1つ（システム全体の権限）
- **プロジェクトロール**: プロジェクトごとに0個以上（プロジェクト単位の権限）

### 6.1.1 プロジェクトメンバーシップ

プロジェクトへの参加は「プロジェクトメンバーシップ」によって管理される。
プロジェクトメンバーシップとは、ユーザとプロジェクトの紐付けであり、プロジェクトごとにプロジェクトロール（PM/TL・メンバー・閲覧者）を個別に設定する仕組みである。

#### 設計意図
同一ユーザが複数プロジェクトに参加する場合、プロジェクトごとに異なるロールを持てるようにする。
これにより、あるプロジェクトではリーダーとして運営責任を担い、別のプロジェクトではメンバーとして作業に従事する、といった柔軟な権限運用が可能になる。

#### 具体例

| ユーザ | プロジェクトA | プロジェクトB | プロジェクトC |
|---|---|---|---|
| Aさん | PM/TL | メンバー | （未参加） |
| Bさん | メンバー | PM/TL | 閲覧者 |
| Cさん | メンバー | メンバー | PM/TL |

#### ルール

- 1ユーザは同一プロジェクトに1つのプロジェクトロールのみ持てる（重複不可）
- プロジェクトに参加していないユーザは、そのプロジェクトの情報にアクセスできない（システム管理者を除く）
- プロジェクトメンバーシップの追加・変更・解除はシステム管理者のみが実行できる

### 6.2 システムロール

#### システム管理者
- ユーザ管理
- システムロール設定/変更
- プロジェクトロール設定/変更
- 全プロジェクト参照
- 必要に応じた代理更新
- 監査ログ参照
- マスタ管理

#### 一般ユーザ
- 自身に付与されたプロジェクトロールに応じた操作のみ可能
- 権限変更不可

### 6.3 プロジェクトロール

#### PM/TL
- プロジェクト運営責任者
- 見積もり、WBS、タスク、リスク、振り返りの管理
- プロジェクト状態遷移

#### メンバー
- 自分の担当タスク更新
- 実績入力
- リスク/課題起票
- ナレッジ下書き登録

#### 閲覧者
- 閲覧のみ

### 6.4 権限変更ルール

- システム管理者のみが、各ユーザのシステムロールおよびプロジェクトロールを設定・変更・解除できる
- 一般ユーザは、自身または他ユーザの権限を変更できない
- 権限変更時は、変更者、変更日時、変更対象、変更前ロール、変更後ロールを記録する
- 権限変更は保存後に即時反映する

### 6.5 ナレッジ公開範囲

- 下書き: 作成者、PM/TL、システム管理者のみ閲覧可
- プロジェクト限定: 当該プロジェクト参加者のみ閲覧可
- 社内公開: 全ログインユーザが閲覧可

### 6.6 削除方針

- 削除は物理削除ではなく論理削除とする
- メンバーおよび閲覧者は削除不可
- PM/TL とシステム管理者のみ削除可

## 8. プロジェクト状態定義

| 状態 | 意味 |
|---|---|
| 企画中 | 案件立上げ、目的・背景・スコープ整理を行う段階 |
| 見積中 | 見積もり作成と根拠整理を行う段階 |
| 計画中 | WBS、タスク、スケジュール、担当割当を確定する段階 |
| 実行中 | 進捗更新、実績記録、リスク/課題対応を行う段階 |
| 完了 | 作業完了後、実績整理と締め処理を行う段階 |
| 振り返り完了 | 振り返りとナレッジ化が完了した段階 |
| クローズ | 最終確定済みで、原則更新不可の段階 |

### 8.1 状態制御の原則
ユーザの操作可否は、ロール権限に加えて、プロジェクト状態によって制限される。
プロジェクト状態による制限はロール権限に優先して適用される。

## 10. 状態遷移時のロック条件

### 10.1 企画中 → 見積中
- 目的、背景、スコープの初版が登録済みであること
- プロジェクト責任者が設定済みであること

### 10.2 見積中 → 計画中
- 見積もりが確定済みであること
- 見積根拠が登録済みであること

### 10.3 計画中 → 実行中
- WBS/タスクが作成済みであること
- 担当者、予定工数、期限が設定済みであること

### 10.4 実行中 → 完了
- 全タスクが完了またはクローズ済みであること
- 主要リスク/課題の状態が整理済みであること
- 実績工数の記録が完了していること

### 10.5 完了 → 振り返り完了
- 振り返りが記録済みであること
- 次回改善事項が登録済みであること
- ナレッジ化対象が整理済みであること

### 10.6 振り返り完了 → クローズ
- 必要なナレッジ公開が完了していること
- プロジェクトの最終状態を確定済みであること

## 10.7 権限ポリシー (2026-04-24 改修)

リスク/課題/振り返り/ナレッジ の 4 エンティジーに共通して以下のポリシーを適用する:

1. **全○○ 画面 (/risks, /issues, /retrospectives, /knowledge)**
   - システム管理者以外: **参照のみ** (作成/編集動線なし)
   - システム管理者: **参照 + 削除のみ** (管理削除用、編集は不可)
2. **各○○一覧 画面 (/projects/[id]/risks|issues|retrospectives|knowledge)**
   - 作成: **実際の ProjectMember** (pm_tl / member) のみ
     - admin でも非メンバーのプロジェクトでは作成不可
   - 編集: **作成者本人のみ** (admin でも他人の記事は編集不可)
   - 削除: **作成者本人** (admin は全○○ から管理削除)
3. **公開種別フィルタ (visibility)**
   - 一覧画面 (全○○ / ○○一覧とも): 非 admin は `visibility='public'` のみ表示
   - admin は `visibility='draft'` も表示 (管理削除のため)
   - 個別 GET: draft は作成者本人 + admin のみ参照可、他は「存在しない」扱い (null)
4. **メモの独立方針**
   - メモは完全な個人資産。CRUD は常に自分のメモのみ、role 判定なし
   - ユーザ削除時はカスケード物理削除 (組織の資産と違い残す意味がない)

詳細は DESIGN.md §8.3 / §8.3.1 / §8.3.2 を参照。

## 12. 共通UI制御ルール

### 12.1 保存系
- 未入力の必須項目がある場合は保存不可または保存時エラーとする
- クローズ状態では保存不可とする
- 振り返り完了後は、対象画面の許容範囲を除き保存不可とする

### 12.2 削除系
- すべて論理削除とする
- メンバー、閲覧者は削除ボタンを表示しない
- クローズ状態では削除不可とする

### 12.3 状態変更系
- プロジェクト状態変更はPM/TL以上のみ可とする
- クローズ解除はシステム管理者のみ可とする
- 状態変更時は遷移条件チェックを必須実施する

### 12.4 公開・確定系
- 見積確定、振り返り確定、ナレッジ公開はPM/TL以上のみ可とする
- メンバーは下書き・提案までとする

### 12.5 実装向け判定式
操作可否は以下で判定する。

`操作可 = ロール可 AND 状態可 AND 対象データ条件可`

例: メンバーが進捗更新できる条件

`メンバーである AND 実行中である AND 自分担当タスクである`

### 13.2 アカウント登録

#### 実行者
システム管理者のみ（一般ユーザによるセルフ登録は不可）

#### 操作手順

1. システム管理者がユーザ管理画面で「新規ユーザ登録」を押下
2. 以下の項目を入力

| 項目 | 必須 | 備考 |
|---|---|---|
| ユーザ名 | ○ | 100文字以内 |
| メールアドレス | ○ | 重複不可 |
| システムロール | ○ | システム管理者 / 一般ユーザ |

※ パスワードは管理者が設定しない。ユーザ自身がパスワード設定画面で設定する。

3. 「招待メールを送信」ボタンを押下
4. 登録されたメールアドレスにパスワード設定リンクを含む招待メールが自動送信される
5. この時点ではアカウントは無効状態（ログイン不可）

#### 13.2.1 初回パスワード設定（ユーザ自身）

##### 一般ユーザ (systemRole='general')

1. ユーザが招待メール内のリンク（/setup-password）をクリック
2. パスワード設定画面でパスワード + 確認パスワードを入力（パスワードポリシー準拠）
3. 設定完了時にリカバリーコード（10個）が画面に1回限り表示される
4. ユーザはリカバリーコードを安全な場所に保管する
5. アカウントが有効化され、ログイン可能になる

##### システム管理者 (systemRole='admin') — **PR #91 で MFA 強制化**

1. ユーザが招待メール内のリンク（/setup-password）をクリック
2. **ステップ 1: パスワード設定** — パスワード + 確認パスワードを入力
   - 成功時: パスワード保存 + MFA シークレット生成 + リカバリーコード作成
   - **アカウントはまだ有効化されない** (isActive=false 維持)
   - **トークンはまだ使用済にしない** (次ステップで使う)
   - 画面は自動的に MFA 登録ステップへ遷移
3. **ステップ 2: MFA 登録** — QR コード表示 + 認証アプリで生成した 6 桁 TOTP を入力
   - 成功時のみ以下を atomic に実行:
     - `emailVerificationToken.usedAt = now`
     - `user.isActive = true`
     - `user.deletedAt = null`
     - `user.mfaEnabled = true`
     - `user.mfaEnabledAt = now`
4. **ステップ 3: リカバリーコード表示** (10 個、1 回限り)
5. ログイン画面に遷移、以後の毎回ログインで MFA 検証が必要

設計意図: 「パスワード + MFA 両方が揃わないと admin アカウントは有効化されない」
ことを技術的に強制する。中断された場合もトークンの有効期限内 (24 時間) なら
同リンクから再開可能 (パスワード設定後なら MFA ステップから)。

#### パスワードポリシー

| ルール | 要件 |
|---|---|
| 最小文字数 | 10文字以上 |
| 文字種 | 英大文字・英小文字・数字・記号のうち3種以上 |
| 最大文字数 | 128文字 |
| 禁止 | メールアドレスと同一、連続同一文字4文字以上 |
| 履歴 | 直近5回のパスワードの再利用不可 |

#### リカバリーコード

| 項目 | 仕様 |
|---|---|
| 形式 | 8文字の英数字 x 10個（例: ABCD-1234） |
| 生成 | 初回パスワード設定時に1回のみ |
| 表示 | パスワード設定完了画面で1回のみ表示。以後再表示不可 |
| 用途 | パスワードリセット時の本人確認、MFA ログイン時のバックアップ |
| 使用回数 | 各コード1回限り |

### 13.3 メール検証（アカウント有効化）

#### フロー

1. ユーザが登録時に送信されたメール内のリンクをクリック
2. システムがトークンを検証

| チェック項目 | 失敗時のメッセージ |
|---|---|
| トークンが存在するか | 「無効なリンクです」 |
| トークンが未使用か | 「既に使用されたリンクです」 |
| 有効期限内か（24時間） | 「有効期限切れです。管理者に再送を依頼してください」 |

3. 全チェック通過でアカウントが有効化される
4. ログイン画面にリダイレクト

#### 検証メール再送
- システム管理者がユーザ管理画面から実行可能
- 前回の送信から5分以上経過していること

#### 未検証アカウントの自動削除
- 7日間検証されなかったアカウントは自動で物理削除される

### 13.4 ログイン

#### 13.4.1 基本ログインフロー（MFA 未設定）

1. ログイン画面でメールアドレスとパスワードを入力
2. システムが以下を順に検証

| チェック項目 | 失敗時の動作 |
|---|---|
| ユーザが存在するか | 汎用エラーメッセージ |
| アカウントが有効か | 汎用エラーメッセージ |
| 恒久ロックされていないか | 汎用エラーメッセージ |
| 一時ロック期間中でないか | 汎用エラーメッセージ |
| パスワードが正しいか | 汎用エラーメッセージ |

※ 全ての失敗に対して同一のメッセージ「メールアドレスまたはパスワードが正しくありません」を表示（ユーザ存在有無の漏洩防止）

3. 認証成功時、セッションが発行されダッシュボードに遷移

#### 13.4.2 MFA 付きログインフロー

1. メールアドレス + パスワードの検証まで同一
2. パスワード検証成功後、MFA 入力画面に遷移（この時点ではセッション未発行）
3. 以下のいずれかで認証を完了

| 方法 | 手順 |
|---|---|
| TOTP コード | 認証アプリに表示される6桁コードを入力 |
| リカバリーコード | 未使用のリカバリーコードを1つ入力（使用後は無効化） |

- TOTP コードの試行回数: 5回まで。超過でステップ1からやり直し
- MFA 入力画面の有効期限: 5分

4. 検証成功でセッション発行、ダッシュボードに遷移

#### 13.4.3 アカウントロックポリシー

| 項目 | 条件 |
|---|---|
| 一時ロック発動 | 10分以内に5回のログイン失敗 |
| 一時ロック期間 | 30分間 |
| 恒久ロック発動 | 一時ロックが3回発生 |
| 恒久ロック解除 | システム管理者が手動で解除 |

#### 13.4.4 ログインエラーメッセージ (PR #87 で差別化)

ログイン失敗時、`POST /api/auth/lock-status` で対象メールのロック状態を参照し、
画面上のエラー文言を 3 パターンで切り替える。ロック中ユーザに「パスワード誤り」と
表示する旧挙動では、ユーザが「何度試してもログインできない」状態に気付けず
問い合わせが増加するため。

| 状況 | 表示メッセージ |
|---|---|
| 永続ロック | アカウントがロックされています。管理者に解除を依頼してください。 |
| 一時ロック (期限内) | ログイン失敗が続いたためアカウントが一時ロックされています。YYYY-MM-DD HH:MM 以降に再度お試しください。 |
| それ以外 (パスワード誤り / 未存在メール 等) | メールアドレスまたはパスワードが正しくありません |

#### 13.4.5 ユーザ削除とカスケード (PR #89 で実装)

| 項目 | 仕様 |
|---|---|
| 削除トリガ | 1) 管理画面「このユーザを削除」ボタン 2) 日次 cron による自動削除 |
| 自動削除条件 | `lastLoginAt` (未ログインの場合 `createdAt`) から 30 日経過 + systemRole='admin' 以外 |
| 削除方式 | User 本体: **論理削除** (deletedAt セット + isActive=false + MFA 無効化) |
| カスケード | ProjectMember / Session / RecoveryCode / EmailVerificationToken / PasswordResetToken / PasswordHistory を**物理削除** |
| 保全対象 | Task.assigneeId / RiskIssue.reporterId / Knowledge.createdBy 等の scalar 参照は残す (履歴保全) |
| 自己削除 | 禁止 (CANNOT_DELETE_SELF) |
| cron 認証 | Vercel Cron + `Authorization: Bearer <CRON_SECRET>` (環境変数) |

**関連**:
- `vercel.json` の `crons` で `/api/admin/users/cleanup-inactive` を日次 03:00 UTC 実行
- `src/config/security.ts` の `INACTIVE_USER_DELETION_DAYS` (既定 30) で閾値変更可

#### 13.4.6 プロジェクト削除カスケードの細粒度制御 (PR #89)

プロジェクト削除ダイアログで「どの資産を削除するか」を 4 つのチェックボックスで個別指定:

| チェック項目 | 対象エンティティ |
|---|---|
| リスク一覧 | `riskIssue` WHERE type='risk' + 紐づく `attachment` |
| 課題一覧 | `riskIssue` WHERE type='issue' + 紐づく `attachment` |
| 振り返り一覧 | `retrospective` + `retrospectiveComment` + 紐づく `attachment` |
| ナレッジ一覧 | `knowledge` (単独紐付けのみ物理削除、共有は紐付け解除のみ) + 紐づく `attachment` |

チェックを入れなかった項目は物理削除されず、全○○ 画面で「親プロジェクト削除済みの孤児資産」として残る。

**強制削除対象 (チェック不能、常に物理削除)**:
プロジェクト本体 / Task + TaskProgressLog (WBS・ガント) / Estimate (見積) / ProjectMember / 上記に紐づく Attachment 全て。

#### 13.4.7 Attachment カスケード削除 (PR #89)

親エンティティ (project / task / estimate / risk / retrospective / knowledge / memo) を削除したとき、
紐づく `attachment` も同時に論理削除する。UI からアクセスできない孤児 attachment が DB に残留するのを防ぐ。

| 親エンティティ | 削除サービス | attachment の削除タイミング |
|---|---|---|
| Task | `deleteTask` | $transaction 内で同時 |
| Estimate | `deleteEstimate` | $transaction 内で同時 |
| Risk/Issue | `deleteRisk` | $transaction 内で同時 |
| Retrospective | `deleteRetrospective` | $transaction 内で同時 |
| Knowledge | `deleteKnowledge` | $transaction 内で同時 |
| Memo | `deleteMemo` | $transaction 内で同時 |
| Project (論理削除) | `deleteProject` | project / task / estimate 配下を同時 |
| Project (物理カスケード) | `deleteProjectCascade` | 削除する全エンティティの attachment を物理削除 |

**enumeration 対策**: `lock-status` API は存在しないメールでも `status: 'none'` を返し、
ロックされていない既存ユーザと区別できないようにしている。ロック状態の開示は、
既に 5 回失敗で自動ロックする挙動自体が観測可能であるため新規の情報漏洩リスクとはならない。

- 上限を超えた場合、最も古いセッションが自動的に無効化される

### 13.8 MFA（多要素認証）設定

#### 13.8.1 MFA 有効化フロー

**一般ユーザ (任意 / 後付け)**:
1. 設定画面で「多要素認証を有効化」を選択
2. QR コードが表示される
3. 認証アプリ（Google Authenticator 等）でスキャンし登録
4. 確認のため TOTP コード（6桁）を1回入力
5. 検証成功で MFA が有効化される

**システム管理者 (必須 / 初期セットアップで強制)** — PR #91 で変更:
- §13.2.1 の初期セットアップフローで **パスワード設定と同時に MFA 登録が必須**
- 設定画面からの手動有効化は不要 (初期セットアップで必ず有効化される)

| 項目 | 仕様 |
|---|---|
| 方式 | TOTP（6桁、30秒間隔） |
| 管理者 | **MFA 強制有効化 (PR #91)** — 初期セットアップで必須、設定画面から解除不可 |
| 一般ユーザ | 任意設定 (従来通り) |

#### 13.8.2 MFA 無効化フロー

**一般ユーザ**:
1. 設定画面で「多要素認証を無効化」を選択
2. MFA が無効化される (TOTP 再確認は現状なし)

**システム管理者**: ❌ **無効化不可** (PR #91)
- 設定画面にボタン自体が表示されない (UI ガード)
- API `POST /api/auth/mfa/disable` が 403 を返す (route ガード)
- サービス層 `disableMfa()` が `CANNOT_DISABLE_ADMIN_MFA` を throw (サービスガード)
- → 3 層の防御で admin が MFA 解除する経路を完全遮断

### 13.13 未使用アカウントの自動削除

#### 段階的な処理

| 段階 | タイミング | 処理 |
|---|---|---|
| 警告メール | 最終ログインから23日後 | 「7日以内にログインしないとアカウントが無効化されます」 |
| 論理削除 | 最終ログインから30日後 | アカウント無効化 + 全セッション無効化 + 通知メール |
| 物理削除 | 論理削除から30日後（最終ログインから60日後） | 個人情報完全削除 |

#### 除外条件
- システム管理者ロールのユーザは自動削除の対象外

#### 論理削除中の復帰
- ユーザがログインすればアカウントが即座に復活する
- プロジェクトメンバーシップは復元されない（管理者が再追加）

#### 物理削除時のデータの扱い

| データ種別 | 処理 |
|---|---|
| 個人情報（氏名・メール） | 完全削除 |
| パスワード・リカバリーコード・MFA 情報 | 完全削除 |
| ユーザが作成したナレッジ・タスク等 | 「削除済みユーザ」名義で保持 |
| 監査ログ・操作ログ | 匿名化して保持（2年間） |

## 16. 核心機能: 提案型サービス (PR #65)

### 16.1 機能概要

本プロダクトの核心価値である「**過去の資産を未来のプロジェクトに自動で活かす**」機能。
人手で過去資産を探す限り発生する抜け漏れを極小化し、未然に防げた課題の顕在化を減らす。
技術詳細は DESIGN.md §23 を参照。

### 16.2 対象資産

| 資産 | 絞り込み | 理由 |
|---|---|---|
| ナレッジ (Knowledge) | `visibility='public'` | 全社共有済の知見のみ候補化 |
| 過去の課題 (Issue) | `type='issue'` + `state='resolved'` + 他プロジェクト | リスクは未発生のため除外、解消済課題のみ過去資産として扱う |

### 16.3 画面・操作

| 画面 / タイミング | 動作 |
|---|---|
| 新規プロジェクト作成直後 | `/projects/[id]?suggestions=1` に遷移 → 提案モーダルが自動起動 |
| プロジェクト詳細「参考」タブ | 常時参照可能、タブ切替で即座に提案リスト表示 |

### 16.4 採用操作

| 対象 | ボタン | 挙動 |
|---|---|---|
| Knowledge | 「このプロジェクトに紐付け」 | `knowledge_projects` に中間レコード追加。両画面で即座に「紐付け済」バッジ表示 |
| 過去 Issue | 「雛形として採用」 | 新規 Issue を type='issue' / state='未対応' / visibility='下書き' で複製。起票者は採用者に差し替え、`result`・`lessonLearned` は継承しない |

### 16.5 権限

| ロール | 閲覧 (GET) | 採用 (POST) |
|---|---|---|
| システム管理者 | 全プロジェクト | 全プロジェクト |
| プロジェクト PM/TL / member | 自プロジェクト | 自プロジェクト |
| 非メンバー | 不可 | 不可 |

### 16.6 精度指標 (内部表示)

提案 1 件ごとに以下の 3 つのスコアを保持し、tooltip で根拠を表示する。
- `tagScore`: タグ Jaccard 係数 (0〜1)
- `textScore`: pg_trgm テキスト類似度 (0〜1)
- `score`: 上記の重み付き平均 (既定 50%/50%)

### 16.7 プロジェクトタグの入力仕様

新規プロジェクト作成時に以下の 3 種類のタグをカンマ区切りで入力できる。
提案精度向上のため入力を推奨する。

| タグ種別 | 例 | マッチング対象 |
|---|---|---|
| 業務ドメイン | 金融 / 基幹業務 / 会計 | Knowledge.businessDomainTags (Phase 2 で対応完了) |
| 技術スタック | React / Next.js / TypeScript / PostgreSQL | Knowledge.techTags |
| 工程 | 要件定義 / 設計 / 開発 / 試験 | Knowledge.processTags |

### 16.8 Phase 2 で追加された機能 (同 PR #65)

**過去振り返りの提示**: プロジェクト詳細「参考」タブ + 新規作成後モーダルに、類似する過去振り返りの問題点・次回事項を参考情報として表示 (採用操作なし)。

**ナレッジ作成時のタグ入力**: ナレッジ作成フォームに 3 種類のタグ入力欄 (業務ドメイン / 技術スタック / 工程) を追加。将来のプロジェクトからの類似検索精度を上げる。

**リスク起票時の inline 提案**: リスク/課題 起票ダイアログで、件名・内容を入力中に 500ms debounce で類似する過去課題 (他プロジェクト) を検索し、警告バナーとして表示。「過去に同種の事象があるので確認を」と気付きを与える UX。

## 17. 一覧列幅のドラッグリサイズ (PR #68)

### 17.1 目的

各一覧/全〇〇画面で長文列 (タスク名 / 件名 / 顧客名 等) が切り詰められ、ユーザが中身を確認できない問題を **UI 上のドラッグ操作**で解消する。

### 17.2 操作

- マウスダウン + ドラッグで任意の幅に変更
- 各一覧の右上にある「列幅をリセット」ボタンで既定値に戻す

### 17.3 対象画面

| 画面 | 対応 |
|---|---|
| `/projects` | 全列リサイズ可 |
| `/risks` / `/issues` | 全列リサイズ可 |

| プロジェクト詳細: リスク/課題/WBS 一覧 | 全列リサイズ可 |

## 19. タスクの状況 / 進捗率 整合性ルール (PR #69 Task 1)

### 19.1 既存ルール (PR #39 から)

- **状況 = 完了 → 進捗率 = 100 を強制**
  - UI のステータス変更時に進捗率を 100 に自動補正
  - サーバ側 (updateTask / bulkUpdateTasks / updateTaskProgress) でも同様に正規化

### 19.2 追加ルール (PR #69)

- **進捗率 = 100 → 状況 = 完了 を強制** (逆方向も双方向に整合)
  - UI で進捗率を 100% に入力した時点で状況が自動で「完了」に切り替わる
  - サーバ側でも同ルールを検証/強制 (不正な "進捗 100% + 状況 = 進行中" 等の組合せを防止)

### 19.3 競合時の優先順位

同一リクエスト内で status と progressRate の両方が矛盾して指定された場合:
- `progressRate=100` と `status≠completed` の組 → **「進捗 100% ならば完了」を優先** (業務ルールとして矛盾する組合せを許容しない)
- `progressRate<100` と `status=completed` の組 → 既存通り「完了 → 進捗 100%」を優先

## 21. 個人メモ機能 (PR #70)

### 21.1 目的

各アカウントが業務上調査して得た知見・作業メモを保存する個人ストック。
既存の Knowledge はプロジェクト紐付け必須で共有前提だったため、
「**プロジェクトに紐付かない個人の知見置き場**」を新設する。

### 21.2 公開範囲 (visibility)

| 値 | 挙動 |
|---|---|
| `private` (既定) | 作成者のみ閲覧可。admin も含め他者は完全に閲覧不可 |
| `public` | 全メモ画面に公開、全ログインユーザが閲覧可 (編集/削除は作成者のみ) |

### 21.3 画面 (PR #71 で 2 画面構成に分離)

| 画面 | URL | 導線 | 表示内容 | 操作 |
|---|---|---|---|---|
| **メモ** (個人管理) | `/memos` | 右上ユーザ名プルダウン → メモ | 自分のメモのみ (private/public 両方) | 作成・編集・削除すべて可能 |
| **全メモ** (横断表示) | `/all-memos` | 上部ナビ (プロジェクト一覧と同列) | 全ユーザの public メモ (自分の公開メモを含む) | 行クリックで**参照専用**ダイアログ (read-only) |

- 両画面とも列幅はリサイズ可 (PR #68 機構、各画面で tableKey を分離)
- 全メモ画面で開く詳細ダイアログは `fieldset disabled` で全入力を読み取り専用化

### 21.4 操作

- **作成**: `/memos` 画面の右上「メモ作成」ボタン (ダイアログ)。全メモ画面には作成ボタンなし
- **編集/削除**: `/memos` 画面でのみ可能 (行クリック or 操作列ボタン)
- **URL 添付**: PR #64 の仕組みを拡張 (entityType='memo')。作成時に staging、編集時に即 CRUD。全メモ画面では閲覧のみ

### 21.5 タグなし

業務知見として有用かは人間の目で判断するため、Knowledge のような tech/process/businessDomain タグは持たせない。
(将来 Knowledge 昇格機能で格上げ時に必要なら別 PR で追加)

### 21.6 認可

- **閲覧**: 作成者 OR visibility='public' のユーザ。**admin 特権なし**
- **作成**: 認証済みユーザなら誰でも
- **編集/削除**: 作成者のみ。**admin 特権なし**
- 他人の private メモにアクセスしようとしても「存在しない」扱い (404) で情報漏洩を防止

### 21.7 API

| メソッド | パス | 認可 | 備考 |
|---|---|---|---|
| GET | /api/memos?scope=mine | 認証済みユーザ | 既定。自分のメモのみ |
| GET | /api/memos?scope=public | 認証済みユーザ | 全ユーザの public メモ (PR #71) |
| POST | /api/memos | 認証済みユーザ | |
| GET | /api/memos/:id | 閲覧権限あるユーザのみ (本人 or public) | |
| PATCH | /api/memos/:id | 作成者のみ | |
| DELETE | /api/memos/:id | 作成者のみ | |

### 21.8 添付 (PR #64 連携)

- `entityType='memo'` を ATTACHMENT_ENTITY_TYPES に追加
- memo は project スコープ外のため、attachment の認可は project membership ではなく memo.userId + visibility で判定
- `authorizeMemoAttachment(memoId, viewerUserId, mode)` を新設 (read/write で権限が異なる)
- batch 取得 API は閲覧可能 memo の ID にフィルタしてから attachments を返す (URL 漏洩防止)

## 22. 日付項目の入力効率化 (PR #71 追加)

### 22.1 変更内容

全ての日付入力項目 (予定開始日 / 終了日・実施日・期限 等) の**右隣**に「**今日**」「**削除**」ボタンを常時表示。

| ボタン | 動作 |
|---|---|
| 今日 | 1 クリックで今日日付 (ローカルタイム YYYY-MM-DD) をセット |
| 削除 | 1 クリックで値をクリア (空欄化)。未入力時は disabled |

必須項目 (required) では「削除」ボタンを非表示にする (空にできない項目で混乱を招かないため)。

### 22.2 導入画面

- プロジェクト新規作成 / 編集 (開始・終了予定日)
- WBS 新規作成 / 個別編集 / bulk edit / bulk actual (予定・実績の各日付)
- 振り返り作成 / 編集 (実施日)
- リスク/課題 編集 (期限)

### 22.3 背景

ブラウザ標準 `<input type="date">` では「今日」「削除」相当操作がカレンダーポップオーバー内部にしかなく、2 ステップ操作 (開く → 探す) が必要だった。ボタン常時露出により 1 ステップ化。

### 22.4 自作カレンダーへの置換 (PR #72)

右側に「今日」「削除」ボタンを外出ししたことで、カレンダーポップアップ内部のベンダー製
「今日」「クリア」ボタンは役割が重複し、UI の一貫性を損ねていた。しかし native `<input type="date">`
ではそれらを CSS / JS で非表示にできない。

よって PR #72 で native input を廃止し、自作の Popover + 日付グリッドに置換した:

- **カレンダー内には月ナビと日付グリッドのみ** (今日 / クリアは出現しない)
- 外側の「今日」「削除」ボタンは PR #71 と同じ仕様で維持
- value / onChange は引き続き `YYYY-MM-DD` 文字列 (呼び出し側 11 箇所に変更なし)

## 23. 画面テーマ設定 (PR #72 追加)

### 23.1 設定画面での選択肢

**設定 → 画面テーマ**に 10 種のラジオ選択肢を配置。

| 大分類 | バリアント |
|---|---|
| ライト / ダーク | ライト (既定) / ダーク |
| パステル (低彩度) | 青 / 緑 / 黄 / 赤 |
| ポップ (高彩度) | 青 / 緑 / 黄 / 赤 |

### 23.2 既定値と永続化

- 既定は **ライトテーマ** (既存挙動と同じ見た目)
- ユーザが変更した値は `users.theme_preference` カラムに保存 → ログイン / ログアウト / ブラウザ再起動を跨いでも維持される
- 未ログイン画面 (ログイン・パスワード再設定等) は常にライトテーマ

### 23.3 反映タイミング

テーマ変更は即時反映 (ページリロード不要):

1. 設定画面で選択
2. `PATCH /api/settings/theme` で DB 更新
3. セッション JWT 更新 + `router.refresh()` でルートレイアウトを再レンダリング
4. `<html data-theme=...>` が書き換わり全画面の配色が切り替わる

### 23.4 セキュリティ

- API サーバは `THEMES` 定数に含まれるキー以外を拒否 (400 VALIDATION_ERROR)
- クライアントからの任意文字列による CSS 注入・データ汚染を防ぐ
- セッション JWT 上のテーマ値を layout 表示前に `toSafeThemeId()` で丸める (二重防御)
