# /knowledge-add — 開発中・後の新ナレッジ追記

KDD (知識駆動開発) フローの Step 4 / Step 6。

## 実行タイミング

**発見時点で即実行**。PR 作成までためない (記憶が揮発する)。

| 発見した内容 | 追記先 |
|---|---|
| 実装中に出会った罠 (型エラー / ハードコード問題 / 設計判断の落とし穴) | `DEVELOPER_GUIDE.md §5.x` |
| 実装パターン・命名規約 | `DEVELOPER_GUIDE.md §5.x` |
| Git/PR/Vercel 関連の運用罠 | `DEVELOPER_GUIDE.md §10.x` |
| E2E test の罠・locator パターン | `E2E_LESSONS_LEARNED.md §4.x` |
| セキュリティ脅威の発見 | `DESIGN.md §脅威` + (重大なら) `/threat-model` |

## ルール (既存運用との整合)

### 1. 重複させない

追記前に必ず以下を実行:

```bash
# 同じキーワードを含む既存セクションを確認
grep -nE "<新セクションのキーワード>" docs/developer/DEVELOPER_GUIDE.md docs/developer/E2E_LESSONS_LEARNED.md
```

既存セクションが見つかった場合の判断:
- **完全に同じ事象**: 既存セクションに「再発事例 N 例目」として追記する (§10.5 方式)
- **似ているが新側面**: 既存セクションをアンカーで参照し、新セクションは差分にフォーカス
- **無関係**: 新セクション作成

### 2. セクション番号の採番

| 文書 | 採番ルール |
|---|---|
| `DEVELOPER_GUIDE.md §5` | 実装パターン、最後の番号 +1 (例: 5.11 の次は 5.12) |
| `DEVELOPER_GUIDE.md §10` | 運用パターン、最後の番号 +1 |
| `E2E_LESSONS_LEARNED.md §4` | E2E 罠、最後の番号 +1 (現状 4.37 まで使用) |

ただし「再発事例」として既存セクションに追加する場合は番号を増やさない。

### 3. テンプレート

#### 新規セクション (DEVELOPER_GUIDE §5 用)

```markdown
### 5.X タイトル (1 行で何の罠/パターンか分かる) (PR/branch 名 で発見)

#### 症状

何が起きたか具体的に (エラーメッセージ・スクリーンショットの引用も歓迎)。

#### 根本原因

なぜ起きたか。記憶ベースの推測ではなく、コード/schema/型定義を Read で確認した上で。

#### 修正

```ts
// 旧
<壊れたコード>

// 新
<直したコード>
```

#### 汎化ルール

1. このパターンを将来避けるための具体ルール
2. 関連箇所の grep コマンド (横展開チェック用)
3. lint/test では検知できない場合の検証手順

#### 関連

- §X.Y (関連する他セクション)
- 関連 PR / commit
```

#### 既存セクションへの再発事例追記

```markdown
#### 再発事例 N 例目 (PR/branch 名 で再発)

直前の §X.Y で「ルール A」を確立したにも関わらず、X という条件で再発:
- (具体的なケース)

**追加教訓**: <ルール A をどう拡張すべきか / 例外条件は何か>
```

### 4. 更新履歴行も必ず追加

`DEVELOPER_GUIDE.md` 末尾の更新履歴テーブルに 1 行追加:

```markdown
| 2026-XX-XX | §5.X 新設 / §X.Y 再発事例 N 例目 (PR #NNN)。<3-5 行で内容> |
```

これにより検索性が上がり、定期 MECE 整理 (`/knowledge-organize`) で
重複検出しやすくなる。

### 5. テスト / lint の更新セット

ナレッジ追加だけでは「次回も同じ罠を踏む」可能性が残るため、可能なら:

- 単体テスト追加 (回帰防止)
- E2E テスト追加 (E2E 系の罠の場合)
- lint rule 追加 (eslint で機械的に検知できる場合)
- TypeScript 型強化 (型システムで検知できる場合)

## 報告フォーマット

```
## /knowledge-add 完了

### 追記先
- <file>:<line> §X.Y <タイトル>

### 既存セクションとの関係
- 重複なし / 既存 §X.Y の再発事例として追記

### 横展開チェック
- 関連箇所 N 件 grep 確認: <結果>

### 更新履歴行
- 追加済 (line NNNN)

### 次のアクション
- なし / 単体テスト追加 / lint rule 追加検討
```
